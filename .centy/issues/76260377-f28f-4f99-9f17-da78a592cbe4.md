# Add wait_for_pane_output domain operation to poll for expected output

After send_keys, there is no domain-level way to wait for the command to produce output. Callers must manually poll capture_pane in a loop with arbitrary sleep intervals, which is error-prone and wasteful.

## Problem
- send_keys is fire-and-forget — no way to know when the command finished
- capture_pane returns a snapshot with no way to wait for new content
- Automation workflows (send command, wait for prompt, send next command) require busy-polling
- No domain concept for "wait until pane output contains X"

## What to do
Add to tmux-gateway-core:

pub struct WaitOptions {
    pub match_text: String,       // text to wait for in pane output
    pub timeout_ms: u32,          // max wait time
    pub poll_interval_ms: u32,    // check interval (default 100ms)
}

pub async fn wait_for_output(target: &str, options: WaitOptions) -> Result<String, TmuxError>

- Repeatedly captures pane output at poll_interval
- Returns the pane content when match_text is found
- Returns TmuxError::Timeout if timeout_ms elapses without a match
- Add a Timeout variant to TmuxError

## Acceptance criteria
- Callers can wait for specific output after send_keys without manual polling
- A timeout prevents unbounded waits
- All API layers can expose this as a blocking or streaming call
