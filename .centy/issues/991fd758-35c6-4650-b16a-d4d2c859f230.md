# Organize domain operations into logical sub-modules as operation count grows

The domain core crate currently has all operations as flat modules in lib.rs. As the operation count grows from 15 to 40+ (with all planned issues), the flat structure will become difficult to navigate and maintain.

## Problem
- lib.rs currently lists 16 modules and 35+ re-exports in a flat structure
- No logical grouping: kill_session sits next to capture_pane sits next to send_keys
- As issues #53-#106 are implemented, the module count will exceed 40
- New contributors will struggle to find related operations

## Proposed solution
Organize operations into logical sub-modules while preserving the public API:

\`\`\`
tmux-gateway-core/src/
├── lib.rs              // re-exports everything (backward compatible)
├── error.rs
├── validation.rs
├── commands.rs         // TmuxCommands trait
├── model/              // domain types
│   ├── session.rs      // TmuxSession, SessionSnapshot
│   ├── window.rs       // TmuxWindow
│   ├── pane.rs         // TmuxPane, CapturePaneResult
│   ├── client.rs       // TmuxClient
│   └── server.rs       // TmuxServerInfo
├── lifecycle/          // create, destroy, rename
│   ├── session.rs      // new_session, kill_session, rename_session
│   ├── window.rs       // new_window, kill_window, rename_window
│   └── pane.rs         // kill_pane, split_window, break_pane, join_pane
├── query/              // read operations
│   ├── list.rs         // list_sessions, list_windows, list_panes, list_clients
│   ├── capture.rs      // capture_pane
│   ├── find.rs         // find_window, display_message
│   └── server.rs       // server_info, show_messages
├── mutation/           // modify state
│   ├── navigation.rs   // select_*, last_*, next_*, previous_*
│   ├── layout.rs       // resize, rotate, swap, move
│   ├── input.rs        // send_keys, pipe_pane
│   └── options.rs      // get/set session/window/pane options
└── admin/              // server-level operations
    ├── server.rs       // kill_server, source_file, run_shell
    ├── environment.rs  // server/session env vars
    ├── hooks.rs        // set_hook, list_hooks
    └── keys.rs         // bind_key, unbind_key, list_keys
\`\`\`

lib.rs continues to re-export everything flat, so existing consumers are unaffected.

## Scope
- Restructure domain crate internal modules
- Preserve all public API exports unchanged
- Group related operations for discoverability
- Update internal imports only
