# Add domain invariant enforcement for tmux structural constraints

tmux enforces certain structural invariants that cause cryptic errors when violated. The domain layer should proactively enforce these constraints with clear error messages rather than passing invalid operations through to tmux.

## Problem
- Killing the last pane in a window kills the window — this implicit behavior should be explicit
- Killing the last window in a session kills the session — same implicit cascade
- Creating a window with a duplicate name succeeds silently in tmux (confusing)
- Splitting a pane below minimum size fails with an unhelpful tmux error
- These constraints are tmux implementation details leaking through the domain

## Proposed solution
Add domain-level invariant checks as a new validation layer:

\`\`\`rust
pub enum DomainInvariant {
    /// Cannot kill the last pane — would implicitly kill the window
    LastPaneInWindow { window_target: String },
    /// Cannot kill the last window — would implicitly kill the session
    LastWindowInSession { session_target: String },
    /// Pane too small to split (below tmux minimum of 2 rows/columns)
    PaneTooSmallToSplit { target: String, dimension: String },
}

#[derive(Debug, thiserror::Error)]
#[error("domain invariant violated: {0}")]
pub struct InvariantViolation(pub DomainInvariant);
\`\`\`

Add invariant checks before destructive operations:
- \`kill_pane\`: check pane count > 1 (or warn about implicit window kill)
- \`kill_window\`: check window count > 1 (or warn about implicit session kill)
- \`split_window\`: check pane dimensions meet minimum

These checks query current state before executing, catching issues at the domain level with clear errors instead of cryptic tmux messages.

## Scope
- New \`invariants.rs\` module in \`tmux-gateway-core\`
- Add \`InvariantViolation\` to error types
- Pre-check logic in relevant operations
- Complements #85 (precondition checks) with structural-specific invariants
