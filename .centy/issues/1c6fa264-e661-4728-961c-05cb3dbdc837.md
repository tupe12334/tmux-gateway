# Split domain functions into pure validation + async infrastructure execution

Each domain function mixes validation, async runtime concerns (tokio::task::spawn_blocking), and tmux infrastructure calls in a single function body. This makes the pure business logic untestable without an async runtime and tmux server.

## Problem
- new_session() validates the name, then immediately spawns a blocking task to call tmux
- list_sessions() parses stdout inside a spawn_blocking closure — parsing logic can not be unit tested in isolation
- Error classification (TmuxError::from_stderr) happens deep inside blocking tasks

## What to do
For each operation, separate into:
1. Pure function: validation + input preparation (no async, no I/O)
2. Parser function: stdout/stderr -> domain types (pure, testable)
3. Infrastructure function: calls tmux and delegates to parser

Example for list_sessions:
- parse_session_line(line: &str) -> Option<TmuxSession> — pure, unit-testable
- classify_tmux_error(stderr: &str, command: &str) -> TmuxError — pure, unit-testable
- list_sessions(backend: &impl TmuxBackend) -> Result<Vec<TmuxSession>, TmuxError> — async shell

## Acceptance criteria
- Parsing logic is tested with plain unit tests (no tokio, no tmux)
- Error classification is tested with canned stderr strings
- spawn_blocking only wraps the minimal tmux I/O call
