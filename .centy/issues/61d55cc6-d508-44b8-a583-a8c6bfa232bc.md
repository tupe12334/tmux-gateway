# Add run-shell domain operation for server-side command execution

tmux \`run-shell\` executes a shell command within the tmux server process context. Unlike \`send-keys\` (which types into a pane), \`run-shell\` runs commands directly and can optionally display output in a target pane. This enables server-side scripting through the gateway.

## Problem
- \`send-keys\` types commands into panes but has no feedback mechanism
- No way to execute commands in the tmux server context (outside any pane)
- Server-side command execution is needed for automation (e.g. setting up hooks, running health checks)
- Missing a key tmux scripting primitive

## Proposed solution

\`\`\`rust
/// Execute a shell command in the tmux server context.
/// If \`target\` is provided, output appears in that pane.
/// Returns the command's stdout output.
pub async fn run_shell(
    command: &str,
    target: Option<&str>,
) -> Result<String, TmuxError>
\`\`\`

Security considerations:
- Command validation is critical — this executes arbitrary shell commands
- API layers MUST gate this with strong authorization
- The domain validates against injection patterns but trusts authorized callers
- Consider a command allowlist at the domain level for defense-in-depth

## Scope
- New \`run_shell.rs\` module in \`tmux-gateway-core\`
- Shell command validation in \`validation.rs\`
- Add to \`TmuxCommands\` trait
- Security documentation for API implementors
