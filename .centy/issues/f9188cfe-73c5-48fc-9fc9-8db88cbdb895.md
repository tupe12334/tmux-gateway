# Add validation to rename_session, rename_window, and new_window domain functions

rename_session, rename_window, and new_window accept raw &str parameters with no validation. Unlike new_session and kill_session which call validate_session_name/validate_session_target, these operations pass unvalidated input directly to tmux.

## Problem
- rename_session(target, new_name): neither target nor new_name is validated
- rename_window(target, new_name): neither validated
- new_window(session, name): neither validated
- A caller could pass shell metacharacters, empty strings, or oversized names
- The command injection protection in validate_target_chars is bypassed for these operations

## What to do
- rename_session: validate target with validate_session_target, validate new_name with validate_session_name
- rename_window: validate target with validate_window_target, validate new_name with validate_session_name (window names follow same rules)
- new_window: validate session with validate_session_target, validate name with validate_session_name
- list_windows: validate session parameter
- list_panes: validate target parameter
- split_window: validate target parameter
- capture_pane: validate target parameter

## Acceptance criteria
- Every domain function validates all string inputs before executing tmux commands
- No unvalidated user input reaches tmux_interface
- Validation errors are returned before any I/O occurs
