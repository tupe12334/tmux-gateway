# Add domain logging port for structured operation-level logging

## Problem

The domain layer (tmux-gateway-core) has zero logging. All logging happens in the API layers (HTTP middleware, gRPC interceptors). This means:

1. Domain-level events are invisible: validation rejections, parsing decisions, operation sequencing
2. Debugging requires correlating API logs with domain behavior by timing
3. Issue #62 proposes tracing spans (instrumentation), #136 proposes metrics — but neither provides human-readable structured logs at the domain level
4. Different log levels for different contexts (verbose for dev, minimal for production) can't be applied at the domain level

## Proposed change

Add a `LogPort` trait in `tmux-gateway-core` following ports & adapters:

```rust
#[derive(Clone, Copy, Debug, PartialEq, Eq)]
pub enum LogLevel {
    Debug,
    Info,
    Warn,
    Error,
}

pub trait LogPort: Send + Sync {
    fn log(&self, level: LogLevel, operation: &str, message: &str);
    fn log_with_target(&self, level: LogLevel, operation: &str, target: &str, message: &str);
}

/// No-op implementation for testing and when logging is disabled
pub struct NoopLog;
impl LogPort for NoopLog {
    fn log(&self, _: LogLevel, _: &str, _: &str) {}
    fn log_with_target(&self, _: LogLevel, _: &str, _: &str, _: &str) {}
}
```

Domain operations use the log port for key events:
- `info`: "creating session 'foo'", "killing window 'bar:0'"
- `warn`: "validation rejected target 'bad;input' — shell metacharacter detected"
- `debug`: "parsed 5 sessions from tmux output (3ms)"
- `error`: "tmux command failed: no server running"

The imperative shell provides the concrete implementation (bridging to `tracing`, `log`, or any framework).

## Why this matters

- Makes domain behavior observable without API-layer coupling
- Complements #62 (tracing spans) with human-readable messages
- Complements #136 (metrics) with qualitative context
- `NoopLog` means zero overhead in tests and when logging is disabled
- API-agnostic: domain logs are the same regardless of which API triggered the operation
- No logging framework dependency in core crate — just a trait

## Acceptance criteria

- [ ] `LogPort` trait with `log()` and `log_with_target()` methods
- [ ] `LogLevel` enum (Debug, Info, Warn, Error)
- [ ] `NoopLog` default implementation
- [ ] At least 5 domain operations instrumented with logging calls
- [ ] Log messages include operation name and target where applicable
- [ ] Unit tests verify NoopLog doesn't panic, trait is object-safe
- [ ] No logging framework dependency in core crate
