# Unify three identical TmuxCommands implementations into a single shared domain service

RestHandler, TmuxGatewayServiceImpl, and the GraphQL schema all implement the TmuxCommands trait with identical bodies — each method just delegates to the corresponding core function. This triple duplication means any change to the delegation pattern must be made in three places.

## Problem

Three files contain identical implementations:
- src/api/rest/routes.rs: RestHandler implements TmuxCommands
- src/api/grpc/tmux_gateway_service.rs: TmuxGatewayServiceImpl implements TmuxCommands
- src/api/graphql/schema.rs: GraphqlHandler implements TmuxCommands (inlined in Query/Mutation)

Each implementation is a verbatim copy:
\`\`\`rust
async fn ls(&self) -> Result<Vec<TmuxSession>, TmuxError> {
    tmux::list_sessions().await
}
async fn create_session(&self, name: &str) -> Result<String, TmuxError> {
    tmux::new_session(name).await
}
// ... 13 identical methods × 3 implementations
\`\`\`

Issue #38 suggests default trait implementations, but the real problem is architectural: there should be one shared service, not three copies.

## Solution

Create a single TmuxService in the domain or application layer:

\`\`\`rust
// src/service.rs or crates/tmux-gateway-core
pub struct TmuxService;

impl TmuxCommands for TmuxService {
    async fn ls(&self) -> Result<Vec<TmuxSession>, TmuxError> {
        tmux::list_sessions().await
    }
    // ... single implementation of all 13 methods
}
\`\`\`

API layers use TmuxService directly instead of implementing the trait themselves:

\`\`\`rust
// REST handler
async fn ls() -> Result<Json<Vec<SessionResponse>>, (StatusCode, String)> {
    let sessions = TmuxService.ls().await.map_err(tmux_err_to_http)?;
    // ... map to response
}
\`\`\`

## Impact

- Eliminates 39 duplicated method implementations (13 methods × 3 handlers)
- Single place to add cross-cutting concerns (logging, metrics, caching)
- API layers focus only on protocol conversion (their actual responsibility)
- Foundation for adding middleware at the domain service level (timeouts, retries, locks)
