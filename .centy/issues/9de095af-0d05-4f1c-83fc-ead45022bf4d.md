# Add command pipeline optimization for batched tmux invocations

Every domain operation spawns a separate tmux process via `exec_tmux`. For composite operations and batch workflows (#93), this means N subprocess invocations for N commands. tmux supports command chaining with `\;` separators, allowing multiple commands in a single tmux invocation with significantly less overhead.

## Problem

- Each domain operation spawns a new subprocess (`Tmux::with_command(...).output()`)
- `create_session_with_windows` spawns N+1 processes for N windows
- Batch operations (#93) would multiply this further
- Subprocess creation has non-trivial overhead (fork, exec, socket connect)
- Multiple sequential commands on the same session could be pipelined
- The `tmux_interface` crate supports command chaining but the domain doesn't use it

## Solution

Add a command pipeline builder in the executor layer:

```rust
pub struct TmuxPipeline {
    commands: Vec<(String, Box<dyn Into<TmuxCommand>>)>,
}

impl TmuxPipeline {
    pub fn new() -> Self { Self { commands: vec![] } }

    pub fn add(mut self, name: &str, cmd: impl Into<TmuxCommand> + 'static) -> Self {
        self.commands.push((name.to_string(), Box::new(cmd)));
        self
    }

    /// Execute all commands in a single tmux invocation using \; chaining
    pub async fn execute(self) -> Result<(), TmuxError> {
        // Build: tmux cmd1 \; cmd2 \; cmd3
        // Single subprocess, single socket connection
    }
}
```

Refactor composite operations to use pipelines:

```rust
pub async fn create_session_with_windows(name: &str, windows: &[&str]) -> Result<TmuxSession, TmuxError> {
    TmuxPipeline::new()
        .add("new-session", NewSession::new().detached().session_name(name))
        .add("rename-window", RenameWindow::new().new_name(windows[0]))
        // ... additional windows
        .execute()
        .await?;
    get_session(name).await
}
```

## Impact

- Reduces subprocess overhead for multi-command operations
- Single tmux connection per pipeline (less socket overhead)
- Enables efficient batch operations (#93)
- Maintains the same domain API — pipeline is an internal optimization
