# Add Dimensions value object for pane geometry with derived computations

## Problem

Pane dimensions are stored as separate `width: u32` and `height: u32` fields in `TmuxPane`. This misses an opportunity to model geometry as a first-class domain concept with useful derived computations.

Operations that depend on geometry (split_window, resize_pane, layout management) need to reason about whether a split is feasible, what the resulting dimensions would be, and whether a pane meets minimum size requirements. Currently this logic doesn't exist in the domain layer.

## Proposed change

Add a `Dimensions` value object in `tmux-gateway-core`:

```rust
#[derive(Clone, Copy, Debug, PartialEq, Eq, Hash)]
pub struct Dimensions {
    width: u32,
    height: u32,
}

impl Dimensions {
    pub fn new(width: u32, height: u32) -> Self;

    // Pure computations
    pub fn width(&self) -> u32;
    pub fn height(&self) -> u32;
    pub fn area(&self) -> u64;
    pub fn aspect_ratio(&self) -> f64;
    pub fn is_portrait(&self) -> bool;
    pub fn is_landscape(&self) -> bool;

    // Split feasibility checks
    pub fn can_split_horizontal(&self, min_pane_height: u32) -> bool;
    pub fn can_split_vertical(&self, min_pane_width: u32) -> bool;
    pub fn after_horizontal_split(&self) -> (Dimensions, Dimensions);
    pub fn after_vertical_split(&self) -> (Dimensions, Dimensions);

    // Size constraints
    pub fn meets_minimum(&self, min_width: u32, min_height: u32) -> bool;
    pub fn contains(&self, other: &Dimensions) -> bool;
}
```

Update `TmuxPane` to use `Dimensions`:
```rust
pub struct TmuxPane {
    pub id: String,
    pub dimensions: Dimensions,
    pub active: bool,
}
```

## Why this matters

- Issue #53 (resize_pane) needs geometry reasoning to validate resize requests
- Issue #69 (layout management) needs dimension computations for layout validation
- Issue #125 (WindowLayout VO) needs pane dimensions for layout tree construction
- Split feasibility checks prevent errors before they reach tmux
- All computations are pure — zero I/O, fully testable with arbitrary dimensions

## Acceptance criteria

- [ ] `Dimensions` struct with all pure computation methods
- [ ] `TmuxPane` updated to use `Dimensions` instead of separate width/height
- [ ] `can_split_*` methods with configurable minimum pane size (tmux default: 2 lines/columns)
- [ ] `after_*_split` methods that compute resulting pane dimensions
- [ ] Unit tests for all computations including edge cases (1x1, very large, zero)
- [ ] Parsing updated to construct `Dimensions` from tmux output
