# Normalize capture_pane output in domain layer

## Problem

`capture_pane()` in `crates/tmux-gateway-core/src/capture_pane.rs` returns the raw stdout from tmux's `capture-pane -p` command via `String::from_utf8_lossy(&output.stdout).to_string()`. This raw output includes:

- Trailing blank lines (tmux pads to full pane height)
- Potential trailing whitespace on each line
- No defined encoding contract (lossy UTF-8 conversion silently replaces invalid bytes)

The domain should define what "captured pane content" means semantically, rather than leaking tmux's raw output format.

## Current state

```rust
let content = String::from_utf8_lossy(&output.stdout).to_string();
Ok(content)
```

## Proposed changes

1. Trim trailing blank lines from the captured output (tmux pads to pane height)
2. Define a `CapturedContent` domain type (or at minimum document the normalization contract) rather than raw String
3. Optionally: trim trailing whitespace per line
4. Consider returning an error rather than silently replacing invalid UTF-8 bytes

## Acceptance criteria

- [ ] capture_pane output has trailing blank lines stripped
- [ ] Normalization is a pure function that can be tested independently
- [ ] Existing API consumers get cleaner output without behavior changes
- [ ] Edge cases tested: empty pane, pane with only whitespace, binary content
