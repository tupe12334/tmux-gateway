# Add pane marking domain operations

## Problem

Tmux has a "marked pane" feature (`select-pane -m` / `select-pane -M`) that marks a single pane globally. The marked pane is used as the implicit source for `swap-pane` and `join-pane` operations. This is a core tmux workflow primitive: mark a source pane, navigate to the destination, then swap.

The gateway has no domain operation for marking/unmarking panes or querying which pane is currently marked. This prevents clients from using the mark-based swap workflow and leaves the `{marked}` target specifier unsupported.

## Proposed change

Add mark/unmark pane domain operations:

```rust
/// Mark the target pane as the "marked pane" (globally, one pane at a time).

    // tmux select-pane -m -t {target}
}

/// Clear the marked pane (unmark).
pub async fn unmark_pane(executor: &impl TmuxExecutor) -> Result<(), TmuxError> {
    // tmux select-pane -M
}

/// Get the currently marked pane, if any.
pub async fn get_marked_pane(executor: &impl TmuxExecutor) -> Result<Option<String>, TmuxError> {
    // Query #{pane_marked} across all panes, return the marked one's ID
}
```

Also add `marked: bool` field to `TmuxPane` (populated from `#{pane_marked}` in `list_panes`).

## Why this matters

- Enables the mark-and-swap workflow that power users rely on
- Supports the `{marked}` target specifier in other operations
- `marked` field on `TmuxPane` gives clients pane state visibility
- Complements `swap_panes` (#93) — marking is the idiomatic way to select a swap source
- API-agnostic: all APIs can expose mark/unmark/query

## Acceptance criteria

- [ ] `mark_pane(executor, target)` marks a pane via `select-pane -m`
- [ ] `unmark_pane(executor)` clears the mark via `select-pane -M`
- [ ] `get_marked_pane(executor)` returns the marked pane's ID or None
- [ ] `TmuxPane` struct includes `marked: bool` field
- [ ] Target validation on mark_pane
- [ ] Unit tests with mock executor
- [ ] Only one pane can be marked at a time (tmux enforces this)
