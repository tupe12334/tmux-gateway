# Add domain-level session and resource limits to prevent unbounded growth

The domain has no concept of resource limits. A caller can create unlimited sessions, windows, and panes until the system runs out of memory or file descriptors. No domain-level policy prevents this.

## Problem
- new_session can be called indefinitely with no ceiling
- new_window and split_window can grow a session without bound
- send_keys has no rate limit at the domain level
- A misbehaving or malicious client can exhaust tmux server resources
- Issue #16 covers API-level rate limiting, but the domain itself has no resource awareness

## What to do
Add configurable domain limits:

pub struct DomainLimits {
    pub max_sessions: u32,          // e.g., 100
    pub max_windows_per_session: u32, // e.g., 50
    pub max_panes_per_window: u32,   // e.g., 16
}

Check limits before executing create/split operations:
- new_session: count current sessions, reject if at max
- new_window: count windows in session, reject if at max
- split_window: count panes in window, reject if at max

Add a new error variant: TmuxError::ResourceLimitExceeded { resource: String, limit: u32 }

## Acceptance criteria
- Domain functions enforce configurable resource limits
- Exceeding limits returns a typed error, not a tmux failure
- Limits are checked before executing tmux commands (fail fast)
- All API layers surface the limit error appropriately
