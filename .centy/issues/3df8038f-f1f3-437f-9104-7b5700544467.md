# Support initial command parameter in create operations

## Problem

The domain's create operations (new_session, new_window, split_window) accept structural parameters (name, target, horizontal) but not an initial command to execute. tmux natively supports launching a process when creating sessions, windows, or panes:

- `tmux new-session -d -s name 'vim file.txt'`
- `tmux new-window -t session 'htop'`
- `tmux split-window -t target 'tail -f log'`

This is a core domain capability — launching processes in tmux entities is one of the primary use cases for a tmux gateway. Without it, clients must create an entity and then use send_keys to type a command, which is fragile and racey.

## Current signatures

```rust
pub async fn new_session(name: &str) -> Result<String, TmuxError>
pub async fn new_window(session: &str, name: &str) -> Result<String, TmuxError>
pub async fn split_window(target: &str, horizontal: bool) -> Result<(), TmuxError>
```

## Proposed changes

Add an optional `command: Option<&str>` parameter to each function:

```rust
pub async fn new_session(name: &str, command: Option<&str>) -> Result<String, TmuxError>
pub async fn new_window(session: &str, name: &str, command: Option<&str>) -> Result<String, TmuxError>
pub async fn split_window(target: &str, horizontal: bool, command: Option<&str>) -> Result<(), TmuxError>
```

- Update TmuxCommands trait signatures
- Add command validation (reject shell metacharacters for safety)
- All API layers pass through the optional parameter

## Acceptance criteria

- [ ] new_session, new_window, split_window accept optional initial command
- [ ] Command parameter is validated for safety (no injection)
- [ ] Passing None preserves current behavior (default shell)
- [ ] TmuxCommands trait updated with new signatures
- [ ] Tests verify command execution in created entities
