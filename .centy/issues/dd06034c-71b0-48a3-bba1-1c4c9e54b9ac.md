# Add domain operation for server-level environment variables

Issue #77 covers session-level environment variables, but tmux also maintains server-level (global) environment variables that are inherited by all new sessions. These are set with \`set-environment -g\` and are a separate scope from session env vars.

## Problem
- No way to get or set global tmux environment variables through the gateway
- Server-level env vars control default behavior for all sessions (e.g. PATH, EDITOR, TERM)
- Session env (#77) and server env are different scopes — both needed for complete environment management
- \`show-environment -g\` and \`set-environment -g\` have no domain representation

## Proposed solution
Add domain operations for server-level environment:

\`\`\`rust
/// Get all server-level environment variables.
pub async fn list_server_environment() -> Result<Vec<EnvVar>, TmuxError>

/// Get a specific server-level environment variable.
pub async fn get_server_env(name: &str) -> Result<Option<String>, TmuxError>

/// Set a server-level environment variable.
pub async fn set_server_env(name: &str, value: &str) -> Result<(), TmuxError>

/// Remove a server-level environment variable.
pub async fn unset_server_env(name: &str) -> Result<(), TmuxError>

pub struct EnvVar {
    pub name: String,
    pub value: String,
}
\`\`\`

Validate environment variable names (alphanumeric + underscores, must start with letter or underscore) and values (no null bytes). Share the \`EnvVar\` model with session environment operations (#77) for consistency.

## Scope
- New \`server_environment.rs\` module in \`tmux-gateway-core\`
- Shared \`EnvVar\` value object usable by both server and session environment operations
- Add env var name/value validation to \`validation.rs\`
- Add to \`TmuxCommands\` trait
- Coordinate with #77 for consistent API design across environment scopes
