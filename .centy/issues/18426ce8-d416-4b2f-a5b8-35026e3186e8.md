# Define domain policy for duplicate session creation (idempotent vs strict)

Creating a session that already exists returns a TmuxError::CommandFailed because tmux itself errors. There is no domain-level decision on whether this should be an error or an idempotent success.

## Problem
- new_session("existing") fails with a tmux error "duplicate session: existing"
- Callers have no way to express "create if not exists" vs "fail if exists"
- In distributed or retry scenarios, duplicate creation errors cause unnecessary failures
- The domain has no concept of idempotency for any operation

## What to do
- Decide on domain semantics:
  - Option A: Add a CreateMode enum: CreateMode::Strict (fail if exists) vs CreateMode::Idempotent (return existing)
  - Option B: Add a separate ensure_session(name) function that creates only if needed
  - Option C: Classify "duplicate session" stderr as a specific DuplicateSession error variant so callers can handle it
- At minimum, add a DuplicateSession variant to TmuxError so callers can distinguish "already exists" from other failures

## Acceptance criteria
- "Session already exists" is a distinct, typed error (not CommandFailed)
- Callers can programmatically distinguish duplicate creation from other errors
- Domain documents its idempotency semantics
