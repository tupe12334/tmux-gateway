# Return richer types from mutating domain operations

## Problem

Several domain operations that create or modify tmux resources return minimal information:

* split_window returns () — caller doesn’t know the new pane’s ID
* new_session returns String (name) — caller doesn’t get full session info
* new_window returns String (name) — caller doesn’t get window index or other details
* kill\_\* operations return () — no confirmation of what was killed

This forces API layers to make follow-up queries if they need details about the result.

## Current state

````rust
pub async fn split_window(target: &str, horizontal: bool) -> Result<(), TmuxError>
pub async fn new_session(name: &str) -> Result<String, TmuxError>
pub async fn new_window(session: &str, name: &str) -> Result<String, TmuxError>
````

## Proposed change

Return domain types from creation operations:

* `split_window` → `Result<TmuxPane, TmuxError>` (returns the newly created pane)
* `new_session` → `Result<TmuxSession, TmuxError>` (returns the created session with all fields)
* `new_window` → `Result<TmuxWindow, TmuxError>` (returns window index, pane count, etc.)

Use tmux format strings in the creation commands to capture output about the new resource, then parse it into the domain type.

## Acceptance criteria

* [ ] split_window returns the new TmuxPane
* [ ] new_session returns the created TmuxSession
* [ ] new_window returns the created TmuxWindow
* [ ] Parsing uses the pure parse functions from #36
* [ ] All API layers updated to use the richer return types
* [ ] No extra tmux round-trips needed
