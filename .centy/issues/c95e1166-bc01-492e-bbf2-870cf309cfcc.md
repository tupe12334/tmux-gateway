# Improve test cleanup reliability with guard-based session cleanup

## Problem

Integration tests that create tmux sessions don’t reliably clean up after themselves if an assertion fails mid-test. For example, `rest_session_lifecycle` and `cross_protocol_*` tests create sessions but only clean up on the happy path. If an assertion panics, the session leaks and can interfere with subsequent test runs.

Additionally, tests silently skip when tmux isn’t available (`if !common::tmux_available() { return; }`), which masks whether tests actually ran in CI.

## Current state

In tests/common/mod.rs:

````rust
pub fn cleanup_session(name: &str) {
    let _ = std::process::Command::new("tmux").args(["kill-session", "-t", name]).output();
}
````

Only `concurrent_session_creation` calls cleanup explicitly. Other tests rely on their own kill operations succeeding.

## Proposed change

1. Add a RAII guard for test sessions:

````rust
pub struct TestSession {
    pub name: String,
}

impl TestSession {
    pub fn new() -> Self {
        Self { name: unique_session_name() }
    }
}

impl Drop for TestSession {
    fn drop(&mut self) {
        cleanup_session(&self.name);
    }
}
````

2. Use `#[ignore]` or panic with a clear message when tmux is unavailable instead of silently returning.

These are domain test quality improvements — not API-specific.

## Acceptance criteria

* [ ] TestSession guard struct with Drop-based cleanup
* [ ] All integration tests use the guard pattern
* [ ] Tests that require tmux fail loudly when it’s unavailable (or are marked #\[ignore\])
* [ ] No leaked sessions after test runs, even on assertion failures
