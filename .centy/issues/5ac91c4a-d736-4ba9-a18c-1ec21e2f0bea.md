# Remove TmuxCommands trait from core — API layers should call domain functions directly

The TmuxCommands trait in tmux-gateway-core is implemented identically in all three API layers (REST, GraphQL, gRPC). Each implementation is pure boilerplate that delegates to the same core functions. The trait adds coupling without domain value.

## Problem
- RestHandler, GraphqlHandler, and TmuxGatewayServiceImpl all implement TmuxCommands with identical method bodies
- Adding a new domain operation requires updating the trait + all three implementations
- The trait lives in the core crate but serves an API-layer concern (polymorphic dispatch across handlers)
- It forces async Future constraints onto the domain layer

## What to do
- Remove TmuxCommands trait from tmux-gateway-core
- API layers call core functions directly: tmux::list_sessions().await, tmux::new_session(name).await
- Remove the three identical handler structs (RestHandler, GraphqlHandler, TmuxGatewayServiceImpl as TmuxCommands impl)
- If polymorphism is needed later (e.g., for mock testing), introduce it at the API layer level, not in the domain

## Acceptance criteria
- Core crate exports only domain functions, not API-layer traits
- Adding a new domain operation requires only adding the function to core — no trait updates
- All API layers work by calling core functions directly
- No duplicate boilerplate implementations
