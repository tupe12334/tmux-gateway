# Add tmux version detection and feature compatibility gating in domain layer

The domain layer has no awareness of the running tmux version. Operations like display-popup (3.2+), pipe-pane flags, and certain format strings differ across tmux versions. The domain blindly sends commands that may not be supported, producing cryptic tmux errors.

## Problem

- No domain concept of tmux server version capabilities
- Operations added for newer tmux features (display-popup, extended formats) will fail silently or with unhelpful errors on older versions
- Clients have no way to discover which operations are available
- The server_info operation returns the version string but it is not used for any gating logic
- Production deployments may run different tmux versions across hosts

## Solution

Add a domain-level version capability model:

```rust
pub struct TmuxVersion {
    pub major: u32,
    pub minor: u32,
}

impl TmuxVersion {
    pub fn supports(&self, feature: TmuxFeature) -> bool { ... }
}

pub enum TmuxFeature {
    DisplayPopup,     // 3.2+
    PipePaneFlags,    // 3.0+
    ExtendedFormats,  // 3.1+
    SessionGroups,    // 1.9+
    HookSystem,       // 2.4+
    WaitFor,          // 1.8+
    CopyMode,         // 2.4+ (programmatic)
}
```

Domain operations should check version support before executing:
```rust
pub async fn display_popup(executor: &dyn TmuxExecutor, ...) -> Result<(), TmuxError> {
    let version = get_server_version(executor)?;
    if !version.supports(TmuxFeature::DisplayPopup) {
        return Err(TmuxError::UnsupportedFeature {
            feature: "display-popup",
            required: "3.2",
            actual: version.to_string(),
        });
    }
    // ... proceed with operation
}
```

## Impact

- Clear error messages when a feature isn't supported by the running tmux version
- Clients can query capabilities before attempting operations
- Prevents silent failures in mixed-version deployments
- Enables graceful degradation for version-dependent features
