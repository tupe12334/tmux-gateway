# Add domain-level output transformation pipeline for pane content

## Problem

Captured pane content often needs transformation before it's useful to consumers: stripping ANSI escape codes, extracting the last N lines, searching for patterns, or trimming trailing blank lines. Currently this transformation burden falls entirely on API consumers.

These are pure string transformations that belong in the domain layer as reusable building blocks — they have no I/O dependencies and are trivially testable.

## Proposed change

Add a `ContentTransform` module in `tmux-gateway-core` with composable pure functions:

```rust
/// Pure: strip all ANSI escape sequences from content
pub fn strip_ansi(content: &str) -> String;

/// Pure: return the last N lines
pub fn tail(content: &str, n: usize) -> String;

/// Pure: return lines matching a pattern (simple substring match)
pub fn grep(content: &str, pattern: &str) -> Vec<(usize, String)>;

/// Pure: remove trailing blank lines
pub fn trim_trailing_blanks(content: &str) -> String;

/// Pure: truncate content to max byte size, respecting line boundaries
pub fn truncate_to_bytes(content: &str, max_bytes: usize) -> (String, bool);

/// Composable pipeline
pub struct ContentPipeline {
    transforms: Vec<Box<dyn Fn(String) -> String>>,
}

impl ContentPipeline {
    pub fn new() -> Self;
    pub fn strip_ansi(self) -> Self;
    pub fn tail(self, n: usize) -> Self;
    pub fn trim(self) -> Self;
    pub fn apply(&self, content: String) -> String;
}
```

## Why this matters

- Issue #116 (ANSI handling) requires `strip_ansi` — this provides the domain foundation
- Issue #75 (normalize capture output) needs `trim_trailing_blanks`
- Issue #115 (change detection) benefits from normalizing content before diffing
- All transformations are pure functions with zero I/O — perfect functional core candidates
- API layers can expose pipeline options (e.g., `?strip_ansi=true&tail=50`) without duplicating logic

## Acceptance criteria

- [ ] `strip_ansi()` handles common escape codes (SGR, cursor movement, erase)
- [ ] `tail()`, `grep()`, `trim_trailing_blanks()`, `truncate_to_bytes()` implemented as pure functions
- [ ] `ContentPipeline` for composing transforms in sequence
- [ ] Comprehensive unit tests for each transform (empty input, single line, ANSI edge cases, UTF-8)
- [ ] No external regex crate dependency — use simple state machine for ANSI stripping
