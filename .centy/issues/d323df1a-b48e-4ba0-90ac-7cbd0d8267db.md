# Add SessionAggregate with recursive loading operation

## Problem

Clients frequently need a session's complete state: the session itself, its windows, and all panes in each window. Currently this requires 1 + N calls (1 for list_windows, N for list_panes per window), orchestrated by each API layer independently.

Issue #74 proposes a hierarchical domain model (relationships between types), but there's no domain operation that actually loads the aggregate — and no type representing the loaded result.

## Proposed change

Add aggregate types and a loading operation in `tmux-gateway-core`:

```rust
#[derive(Clone, Debug, PartialEq)]
pub struct PaneSnapshot {
    pub pane: TmuxPane,
}

#[derive(Clone, Debug, PartialEq)]
pub struct WindowAggregate {
    pub window: TmuxWindow,
    pub panes: Vec<PaneSnapshot>,
}

#[derive(Clone, Debug, PartialEq)]
pub struct SessionAggregate {
    pub session: TmuxSession,
    pub windows: Vec<WindowAggregate>,
}

impl SessionAggregate {
    // Pure query methods on the loaded aggregate
    pub fn total_pane_count(&self) -> usize;
    pub fn active_window(&self) -> Option<&WindowAggregate>;
    pub fn find_window_by_name(&self, name: &str) -> Option<&WindowAggregate>;
    pub fn find_pane_by_id(&self, id: &str) -> Option<&PaneSnapshot>;
    pub fn total_area(&self) -> u64;
}
```

Loading operation:
```rust
pub async fn load_session_aggregate(session_name: &str) -> Result<SessionAggregate, TmuxError>;
```

This calls `list_windows` then `list_panes` for each window, assembling the aggregate. The aggregate itself is a pure value with query methods.

## Why this matters

- Eliminates N+1 query patterns in all API layers
- Foundation for #107 (snapshot/restore) — a snapshot is essentially a serialized aggregate
- Foundation for #138 (session topology stats) — stats can be computed from aggregates
- GraphQL resolvers can return nested data without multiple tmux calls
- Pure query methods on the aggregate enable issue #135 (query combinators) on hierarchical data
- The loading operation is the only I/O part; all query methods are pure

## Acceptance criteria

- [ ] `SessionAggregate`, `WindowAggregate`, `PaneSnapshot` types in core crate
- [ ] `load_session_aggregate()` async operation that recursively loads
- [ ] Pure query methods: `total_pane_count`, `active_window`, `find_window_by_name`, `find_pane_by_id`
- [ ] Unit tests for all pure methods on hand-constructed aggregates
- [ ] Integration test that loads a real session aggregate (skip if tmux unavailable)
- [ ] Error handling for partial load failures (some windows succeed, some fail)
