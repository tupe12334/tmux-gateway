# Extract pure validate-build-parse pipeline for each domain operation

Each domain operation currently mixes validation, command construction, I/O execution, and output parsing in a single async function. This makes the logic hard to test in isolation and violates the functional core / imperative shell pattern.

## Problem

Current pattern in every operation (e.g., list_windows.rs):
```rust
pub async fn list_windows(session: &str) -> Result<Vec<TmuxWindow>, TmuxError> {
    validation::validate_session_name(session)?;       // 1. validate
    let session = session.to_string();
    tokio::task::spawn_blocking(move || {
        let output = Tmux::with_command(                // 2. build + execute (mixed)
            ListWindows::new()
                .target_session(session.as_str())
                .format("#{window_index}\t...")
        ).output()?;
        // 3. parse (inside I/O closure)
        for line in stdout.lines() {
            let parts: Vec<&str> = line.splitn(4, '\t').collect();
            // ...
        }
    }).await?
}
```

The parsing logic is buried inside spawn_blocking and cannot be tested without executing tmux.

## Solution

Decompose each operation into three pure functions + one imperative orchestrator:

```rust
// Pure: validate input
fn validate_list_windows_input(session: &str) -> Result<SessionName, ValidationError>;

// Pure: build the tmux command specification
fn build_list_windows_command(session: &SessionName) -> TmuxCommandSpec;

// Pure: parse raw output into domain types
fn parse_list_windows_output(stdout: &str) -> Result<Vec<TmuxWindow>, ParseError>;

// Imperative shell: orchestrate I/O
pub async fn list_windows(session: &str) -> Result<Vec<TmuxWindow>, TmuxError> {
    let session = validate_list_windows_input(session)?;
    let spec = build_list_windows_command(&session);
    let output = execute_tmux(spec).await?;
    parse_list_windows_output(&output)
}
```

## Impact

- Parse functions become unit-testable without tmux (pure fn with string input/output)
- Command construction testable without execution
- Validation testable independently (already partially done)
- Clear functional core (validate, build, parse) with imperative shell (execute)
- Consistent pattern across all 15 domain operations
