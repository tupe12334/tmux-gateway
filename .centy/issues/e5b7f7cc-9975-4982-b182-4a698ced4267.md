# Add wait-for domain operation for channel-based synchronization

tmux \`wait-for\` provides named channels for inter-process synchronization. A process can block on \`wait-for channel\` until another signals it with \`wait-for -S channel\`. This is essential for orchestrating multi-step workflows where operations must be sequenced across panes.

## Problem
- No way to synchronize operations across panes through the gateway
- Batch operations (#93) execute sequentially but cannot wait for pane-side events
- Orchestration patterns (e.g. "send command, wait for completion, then continue") require synchronization primitives
- Missing a core tmux feature used in scripting and automation

## Proposed solution

\`\`\`rust
/// Signal a named channel, unblocking any waiters.
pub async fn signal_wait(channel: &str) -> Result<(), TmuxError>

/// Lock a named channel (set it to a locked/waiting state).
pub async fn lock_wait(channel: &str) -> Result<(), TmuxError>

/// Unlock a named channel (release the lock without signaling).
pub async fn unlock_wait(channel: &str) -> Result<(), TmuxError>
\`\`\`

Note: The blocking \`wait-for\` (without -S) is NOT exposed as a domain operation because it would block the async runtime. Instead, only the signaling/locking side is exposed, which is non-blocking. Clients orchestrate waiting externally.

Validate channel names: alphanumeric + hyphens + underscores, max 128 chars.

## Scope
- New \`wait_for.rs\` module in \`tmux-gateway-core\`
- Channel name validation in \`validation.rs\`
- Add signal/lock/unlock to \`TmuxCommands\` trait
- Complements #93 (batch operations) for advanced orchestration
