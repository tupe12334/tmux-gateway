# Add ANSI escape code handling options to capture_pane domain operation

Issue #51 adds line range and scroll history options to capture_pane, but the domain has no concept of output format handling. Pane content contains ANSI escape codes (colors, cursor movement, etc.) that clients may want stripped, preserved, or converted. This is a domain-level output formatting concern.

## Problem

- capture_pane returns raw tmux output which may contain ANSI escape codes
- Clients processing pane output must implement their own ANSI stripping
- tmux capture-pane supports -e (include escapes) and default (strip escapes) modes, but the domain doesn't expose this choice
- Different consumers need different formats: terminal UIs want escapes, log processors want plain text, search operations want stripped text
- The -p flag (to stdout) behavior with escape codes is version-dependent

## Solution

Add output format options to the capture domain:

```rust
pub enum CaptureFormat {
    /// Strip all ANSI escape codes, return plain text (default)
    PlainText,
    /// Preserve ANSI escape codes as-is
    WithEscapes,
}

pub struct CaptureOptions {
    pub format: CaptureFormat,
    pub start_line: Option<i32>,   // From #51
    pub end_line: Option<i32>,     // From #51
    pub include_history: bool,     // From #51
}

pub async fn capture_pane_with_options(
    target: &str,
    options: &CaptureOptions,
) -> Result<PaneContent, TmuxError> {
    validate_pane_target(target)?;
    // Map format option to tmux flags (-e for WithEscapes)
    // ...
}
```

## Impact

- Clients choose the output format appropriate for their use case
- Domain expresses output formatting as a first-class concept
- Reduces client-side processing burden
- Consistent behavior across tmux versions
