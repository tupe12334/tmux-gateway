# Add domain events for session lifecycle changes

Operations like session creation, deletion, window splitting, and key sending produce no observable side effects beyond the tmux state change. There is no audit trail, notification hook, or event stream for consumers to react to state changes.

## Problem
- No way to know what operations happened and when (audit/compliance)
- WebSocket streaming (issue #8) will need an event source — currently none exists
- No hook point for future features like: webhook notifications, session activity logging, usage metrics

## What to do
Define domain events in the core crate:

enum DomainEvent {
    SessionCreated { name: SessionName, timestamp: DateTime<Utc> },
    SessionKilled { target: SessionTarget, timestamp: DateTime<Utc> },
    WindowCreated { session: SessionName, window_name: String, timestamp: DateTime<Utc> },
    WindowKilled { target: WindowTarget, timestamp: DateTime<Utc> },
    PaneKilled { target: PaneTarget, timestamp: DateTime<Utc> },
    KeysSent { target: String, key_count: usize, timestamp: DateTime<Utc> },
    SessionRenamed { old_name: SessionTarget, new_name: String, timestamp: DateTime<Utc> },
}

Domain functions return events alongside their results. An EventBus or channel broadcasts events to interested subscribers (logging, metrics, WebSocket, webhooks).

## Acceptance criteria
- Every mutating domain operation emits an event
- Events are defined in the core crate (API-agnostic)
- An in-process event bus exists for subscribers
- At minimum, events are logged at info level
