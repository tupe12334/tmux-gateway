# Add default TmuxCommands implementation to reduce API boilerplate

## Problem
All three API layers (REST RestHandler, GraphQL QueryRoot/MutationRoot, gRPC TmuxGatewayService) implement the TmuxCommands trait with identical bodies — each method simply delegates to the corresponding free function in tmux-gateway-core. This is pure boilerplate that must be kept in sync across 3 implementations.

## Current state
REST (routes.rs), GraphQL (schema.rs), and gRPC (tmux_gateway_service.rs) each have a struct that implements TmuxCommands with ~13 methods, all forwarding to the same free functions (tmux::list_sessions, tmux::new_session, etc.).

## Proposed change
Add a default implementation to TmuxCommands in the core crate:

```rust
pub trait TmuxCommands {
    async fn ls(&self) -> Result<Vec<TmuxSession>, TmuxError> {
        list_sessions().await
    }
    async fn create_session(&self, name: &str) -> Result<String, TmuxError> {
        new_session(name).await
    }
    // ... default impls for all methods
}
```

API layers can then simply:
```rust
struct RestHandler;
impl TmuxCommands for RestHandler {}
```

Any layer that needs custom behavior (e.g., caching, metrics) can override specific methods.

## Acceptance criteria
- [ ] TmuxCommands has default implementations for all methods
- [ ] All 3 API handler structs use the defaults (empty impl blocks)
- [ ] Any API layer can still override individual methods if needed
- [ ] No behavior change
