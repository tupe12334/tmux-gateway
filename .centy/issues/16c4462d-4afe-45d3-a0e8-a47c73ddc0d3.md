# Add metrics port trait for domain-level operation observability

## Problem

The domain layer has no mechanism for recording operational metrics. All measurement happens at the API layer (HTTP middleware, gRPC interceptors), which means:

1. Domain-internal timings (tmux command latency, parsing duration) are invisible
2. Error type distribution (how many SessionNotFound vs CommandFailed?) is unmeasured
3. Operation throughput (creates/sec, kills/sec) requires each API layer to track independently
4. Issue #62 covers tracing spans but not quantitative metrics

## Proposed change

Add a `MetricsPort` trait in `tmux-gateway-core` following the ports & adapters pattern:

```rust
use std::time::Duration;

pub trait MetricsPort: Send + Sync {
    fn record_operation_duration(&self, operation: &str, duration: Duration);
    fn record_operation_success(&self, operation: &str);
    fn record_operation_failure(&self, operation: &str, error_type: &str);
    fn record_tmux_command_latency(&self, command: &str, duration: Duration);
    fn record_parse_duration(&self, entity_type: &str, duration: Duration);
}

/// No-op implementation for when metrics are not needed
pub struct NoopMetrics;
impl MetricsPort for NoopMetrics {
    fn record_operation_duration(&self, _: &str, _: Duration) {}
    fn record_operation_success(&self, _: &str) {}
    fn record_operation_failure(&self, _: &str, _: &str) {}
    fn record_tmux_command_latency(&self, _: &str, _: Duration) {}
    fn record_parse_duration(&self, _: &str, _: Duration) {}
}
```

Domain operations accept `&dyn MetricsPort` (or the port is injected into the executor). The imperative shell provides the concrete implementation (Prometheus, StatsD, OpenTelemetry, or NoopMetrics for testing).

## Why this matters

- Production-critical: you can't optimize what you can't measure
- Domain-level metrics reveal bottlenecks invisible to API-layer metrics (e.g., tmux parsing taking 50ms)
- Follows hexagonal architecture: domain defines the port, infrastructure provides the adapter
- NoopMetrics means zero overhead when metrics aren't configured
- API-agnostic: metrics flow from domain regardless of whether request came via REST, gRPC, or GraphQL

## Acceptance criteria

- [ ] `MetricsPort` trait in core crate with methods for duration, success, failure, and latency
- [ ] `NoopMetrics` default implementation
- [ ] At least 3 domain operations instrumented with metrics recording
- [ ] Metrics port is optional (default to NoopMetrics if not provided)
- [ ] Unit tests verify NoopMetrics doesn't panic, trait is object-safe
- [ ] No concrete metrics library dependency in core crate
