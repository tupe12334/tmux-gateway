# Add pane output diffing for structured change detection between snapshots

Issue #115 adds content change detection via hashing, but clients monitoring pane output often need to know *what* changed, not just *that* it changed. A structured diff between two pane snapshots enables efficient incremental updates and change visualization.

## Problem

- Change detection (#115) returns boolean "changed or not" — no detail on what changed
- Clients polling for output changes must diff full content client-side
- No domain concept of incremental content updates
- Monitoring dashboards cannot highlight new output lines
- Streaming (#8) handles real-time, but polling clients need efficient diffs

## Solution

Add a domain operation for structured pane content diffing:

```rust
pub struct ContentDiff {
    pub added_lines: Vec<(u32, String)>,     // (line_number, content)
    pub removed_lines: Vec<(u32, String)>,
    pub changed: bool,
    pub old_hash: u64,
    pub new_hash: u64,
}

/// Compute diff between current pane content and a previous snapshot
pub async fn diff_pane_content(
    target: &str,
    previous_content: &str,
) -> Result<ContentDiff, TmuxError> {
    validate_pane_target(target)?;
    let current = capture_pane(target).await?;
    Ok(compute_line_diff(previous_content, &current))
}

/// Compute diff between two content strings (pure function)
pub fn compute_line_diff(old: &str, new: &str) -> ContentDiff
```

## Impact

- Enables efficient incremental pane output monitoring
- Clients receive only changed lines instead of full content
- Reduces bandwidth for high-frequency polling use cases
- Pure diffing function is testable without tmux
- Complements change detection (#115) and content search (#138)
