# Add has-session domain query for lightweight existence checks

There is no way to check if a session exists without calling `list_sessions()` and scanning the full list. The tmux `has-session` command provides an O(1) existence check that returns exit code 0 if the session exists.

## Problem
- Checking session existence requires `list_sessions()` → filter by name — wasteful for large session counts
- Idempotent create patterns (#67) need a fast existence check as a building block
- Health checks and pre-flight validations would benefit from a lightweight probe

## Proposed solution
Add a `has_session` domain operation:

```rust
pub async fn has_session(target: &str) -> Result<bool, TmuxError>
```

- Returns `Ok(true)` if the session exists, `Ok(false)` if not
- Returns `Err(TmuxError::TmuxNotRunning)` if the server isn't running
- Uses tmux `has-session -t <target>` under the hood — no output parsing needed, just exit code
- Validates target with `validate_session_target()` before executing

This is a pure domain query that any API layer can expose (REST HEAD endpoint, GraphQL boolean field, gRPC unary call).

## Scope
- New `has_session.rs` module in `tmux-gateway-core`
- Add to `TmuxCommands` trait
- Export from `lib.rs`
