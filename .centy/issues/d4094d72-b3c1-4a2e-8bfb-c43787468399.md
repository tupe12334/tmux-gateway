# Extract tmux output parsing into pure functions

## Problem

In sessions.rs, list_panes.rs, and list_windows.rs, the tmux stdout parsing logic is embedded inside spawn_blocking closures alongside I/O operations. This violates the functional core / imperative shell principle — parsing is a pure computation that should be separated from the I/O that produces the raw output.

## Current state

Each list function (list_sessions, list_windows, list_panes) contains an inline closure that:

1. Calls tmux via tmux_interface (I/O — imperative shell)
1. Parses tab-delimited stdout into domain types (pure logic — functional core)

These two concerns are interleaved in a single closure, making the parsing logic untestable without running tmux.

## Proposed change

Extract parsing into standalone pure functions in each module:

* `fn parse_session_line(line: &str) -> Option<TmuxSession>`
* `fn parse_window_line(line: &str) -> Option<TmuxWindow>`
* `fn parse_pane_line(line: &str) -> Option<TmuxPane>`

And higher-level:

* `fn parse_sessions(stdout: &str) -> Vec<TmuxSession>`
* `fn parse_windows(stdout: &str) -> Vec<TmuxWindow>`
* `fn parse_panes(stdout: &str) -> Vec<TmuxPane>`

The async list\_\* functions should call tmux, then pass stdout to the pure parser. This enables comprehensive unit testing of parsing without tmux installed.

## Acceptance criteria

* [ ] Each parse function is a standalone `pub(crate) fn` with no I/O dependencies
* [ ] Unit tests cover: valid lines, malformed lines, empty input, lines with missing fields
* [ ] The async list\_\* functions are thin wrappers: tmux I/O → parse → return
* [ ] No behavior change in the public API
