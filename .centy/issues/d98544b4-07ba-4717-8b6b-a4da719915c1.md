# Add PaneContent value object with capture metadata

## Problem

`capture_pane` currently returns a raw `String`. This discards important context: when was the content captured, what were the pane dimensions at capture time, how many lines of scroll history were included, and whether the content was truncated.

Downstream consumers (APIs, snapshot/restore, change detection) need this metadata to make correct decisions.

## Proposed change

Add a `PaneContent` value object in `tmux-gateway-core`:

```rust
pub struct PaneContent {
    content: String,
    target: String,
    captured_at: i64,         // Unix timestamp
    width: u32,
    height: u32,
    line_count: usize,
    includes_history: bool,
    truncated: bool,
}
```

Pure methods on `PaneContent`:

- `lines() -> impl Iterator<Item = &str>` — iterate content lines
- `last_n_lines(n: usize) -> Vec<&str>` — tail of content
- `search(pattern: &str) -> Vec<(usize, &str)>` — find matching lines with line numbers
- `is_empty() -> bool` — true if content is blank/whitespace-only
- `byte_size() -> usize` — content size for transport budgeting

Modify `capture_pane` domain operation to return `PaneContent` instead of `String`. The API layers extract what they need from the value object.

## Why this matters

- Issue #115 (change detection) needs timestamps and dimensions to compute meaningful diffs
- Issue #92 (CapturePaneResult) calls for structured metadata — this provides the domain foundation
- Issue #116 (ANSI handling) needs to know dimensions to correctly strip escape codes
- Snapshot/restore (#107) needs to know if content was truncated

## Acceptance criteria

- [ ] `PaneContent` struct with all fields, derives `Clone, Debug, PartialEq`
- [ ] Pure helper methods with unit tests
- [ ] `capture_pane` returns `PaneContent` instead of `String`
- [ ] Existing API layers updated to extract `.content` for backwards compatibility
- [ ] Unit tests for edge cases: empty capture, single line, very long content
