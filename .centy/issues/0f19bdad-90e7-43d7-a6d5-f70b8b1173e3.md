# Add TmuxServerInfo domain type for tmux health and version

## Problem
Tmux availability checking is duplicated across the codebase and lives outside the domain layer. REST routes.rs has check_tmux_available(), main.rs uses it for gRPC health reporting, and preflight.rs runs its own tmux version check. There is no domain-level concept for "tmux server info" — each layer reinvents the check.

## Current state
- rest/routes.rs: `check_tmux_available()` returns bool, directly calls tmux_interface
- main.rs line 127: calls `rest::check_tmux_available()` for gRPC health — REST module used by gRPC (wrong dependency direction)
- preflight.rs: parses tmux version string independently

## Proposed change
Add a domain type and function in tmux-gateway-core:

```rust
pub struct TmuxServerInfo {
    pub version: String,
    pub running: bool,
}

pub async fn server_info() -> TmuxServerInfo { ... }
pub async fn is_available() -> bool { ... }
```

All API layers and startup code use these domain functions instead of rolling their own checks. This eliminates the cross-API dependency (REST function called from gRPC health setup).

## Acceptance criteria
- [ ] TmuxServerInfo type and is_available/server_info functions in core crate
- [ ] REST health endpoint uses domain function
- [ ] gRPC health setup uses domain function (not rest::check_tmux_available)
- [ ] Preflight uses domain function
- [ ] No cross-API-layer dependencies for tmux health checking
