# Add pane content change detection for efficient polling

Clients that need to monitor pane output must repeatedly call capture_pane and compare the full content. This is wasteful when output hasn't changed. The domain should provide an efficient change detection mechanism.

## Problem

- capture_pane returns the full pane content every time, even if nothing changed
- Polling clients waste bandwidth and processing on unchanged content
- No domain concept of content versioning or change notification
- WebSocket streaming (#8) helps for real-time, but polling clients remain common (cron jobs, monitoring scripts, CI integrations)
- The gateway has no way to tell a client "nothing changed since your last request"

## Solution

Add a content fingerprint to pane capture results:

```rust
pub struct PaneContentSnapshot {
    pub content: String,
    pub content_hash: u64,       // Fast hash of content (e.g., xxhash)
    pub sequence: u64,           // Monotonic counter, incremented on change
    pub captured_at: i64,        // Timestamp of capture
}

/// Check if pane content has changed since a given hash
pub async fn has_pane_changed(
    target: &str,
    last_known_hash: u64,
) -> Result<bool, TmuxError> {
    let current = capture_pane_snapshot(target).await?;
    Ok(current.content_hash != last_known_hash)
}

/// Capture pane content only if changed, returning None if unchanged
pub async fn capture_pane_if_changed(
    target: &str,
    last_known_hash: u64,
) -> Result<Option<PaneContentSnapshot>, TmuxError> {
    let snapshot = capture_pane_snapshot(target).await?;
    if snapshot.content_hash == last_known_hash {
        Ok(None)
    } else {
        Ok(Some(snapshot))
    }
}
```

## Impact

- Dramatically reduces bandwidth for polling clients (304-like semantics)
- Domain-level concept of content versioning
- API layers can implement conditional responses (ETag/If-None-Match for REST, caching for GraphQL)
- Complements WebSocket streaming (#8) for non-streaming use cases
