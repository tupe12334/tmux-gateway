# Add graceful server lifecycle orchestration with snapshot-aware restart

Issue #89 adds `kill-server` and #107 adds session snapshot/restore, but there is no orchestrated lifecycle operation that combines them. In production, the gateway needs to gracefully restart the tmux server (for upgrades, recovery, or maintenance) without losing running sessions.

## Problem

- `kill-server` (#89) destroys all sessions with no preservation
- `snapshot` (#107) captures state but doesn't orchestrate the restart flow
- No way to perform a zero-downtime tmux server restart through the gateway
- Server maintenance requires manual snapshot-kill-restart-restore workflow
- Partial failures during the lifecycle (snapshot succeeds, kill succeeds, restore fails) leave the system in an inconsistent state

## Solution

Add an orchestrated lifecycle operation as a domain-level workflow:

```rust
pub enum LifecycleAction {
    /// Snapshot all sessions, kill server, wait for restart, restore sessions
    GracefulRestart,
    /// Snapshot all sessions, kill server (no restart)
    GracefulShutdown,
    /// Kill server immediately, no snapshot
    ForceShutdown,
}

pub struct LifecycleResult {
    pub sessions_captured: u32,
    pub sessions_restored: u32,
    pub failed_restores: Vec<(String, TmuxError)>,
    pub total_duration: Duration,
}

pub async fn manage_server_lifecycle(
    action: LifecycleAction,
) -> Result<LifecycleResult, TmuxError> {
    match action {
        LifecycleAction::GracefulRestart => {
            let snapshot = snapshot_all_sessions().await?;
            kill_server().await?;
            wait_for_server_ready().await?;
            let result = restore_sessions(&snapshot).await;
            // Report partial failures without aborting
            Ok(result)
        }
        // ...
    }
}
```

## Impact

- Enables zero-downtime tmux server maintenance through the gateway
- Orchestrates snapshot/kill/restore with proper error handling
- Reports partial failures (some sessions restored, some failed)
- Builds on #89 (kill-server) and #107 (snapshot/restore) as primitives
- Essential for production deployments requiring server upgrades
