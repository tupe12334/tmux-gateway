# Add display-panes domain operation for pane identification

## Problem

Tmux's `display-panes` command briefly shows pane indices overlaid on each pane, which is essential for interactive identification of pane targets. The gateway has no corresponding domain operation, meaning clients that render tmux layouts (web UIs, TUI dashboards) cannot trigger this visual aid.

While `display-panes` is inherently visual (it renders in the terminal), the gateway can still expose it as a domain operation for clients that embed or proxy tmux terminal output. Additionally, the underlying pane index/position information can be exposed as structured data.

## Proposed change

Add domain operations for pane identification:

```rust
/// Trigger tmux display-panes on the target window.
/// This causes tmux to briefly overlay pane indices in the terminal.
/// Primarily useful when clients are rendering tmux output (e.g., via capture_pane with escape sequences).
pub async fn display_panes(executor: &impl TmuxExecutor, target: &str) -> Result<(), TmuxError> {
    // Validates target, then executes: tmux display-panes -t {target}
}

/// Get structured pane geometry for a window — positions, sizes, and indices.
/// Useful for clients that render their own pane layout visualization.
pub struct PaneGeometry {
    pub id: String,
    pub index: u32,
    pub x: u32,
    pub y: u32,
    pub width: u32,
    pub height: u32,
    pub active: bool,
}

pub async fn get_pane_geometry(
    executor: &impl TmuxExecutor,
    target: &str,
) -> Result<Vec<PaneGeometry>, TmuxError>
```

The `get_pane_geometry` operation queries tmux format variables (`pane_top`, `pane_left`, `pane_width`, `pane_height`, `pane_index`) to provide structured layout data.

## Why this matters

- Completes pane introspection: `list_panes` gives metadata, `capture_pane` gives content, this gives spatial layout
- Enables web/TUI clients to render accurate pane layouts without guessing positions
- `PaneGeometry` is a pure value object — fits the DDD style perfectly
- API-agnostic: GraphQL clients can query exactly the geometry fields they need

## Acceptance criteria

- [ ] `display_panes(executor, target)` triggers `tmux display-panes`
- [ ] `PaneGeometry` value object with id, index, x, y, width, height, active
- [ ] `get_pane_geometry(executor, target)` parses tmux format output into `Vec<PaneGeometry>`
- [ ] Target validation before execution
- [ ] Unit tests with mock executor for geometry parsing
- [ ] Handles edge case of single-pane window (one geometry entry covering full window)
