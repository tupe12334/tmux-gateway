# Add API versioning strategy and deprecation headers

The API has no versioning mechanism. When breaking changes are needed, there is no way to maintain backward compatibility for existing clients.

## What to do
- Choose a versioning strategy: URL prefix (/v1/), Accept header, or custom header
- Add version prefix to all REST routes (e.g., /v1/ls, /v1/new)
- Keep unversioned routes as aliases for the latest version during transition
- Add Deprecation and Sunset response headers when endpoints are deprecated
- Document the versioning policy in README

## Acceptance criteria
- All endpoints are accessible via /v1/ prefix
- Unversioned routes still work (redirect or alias)
- OpenAPI spec reflects the versioned paths
