# Add SessionAlreadyExists error variant for duplicate detection

## Problem
When creating a session with a name that already exists, tmux rejects it with a stderr message like "duplicate session: foo". The domain layer classifies this as a generic CommandFailed error via from_stderr(), losing the semantic meaning. The domain should have a dedicated error variant for this common case.

## Current state
In error.rs, from_stderr() has no branch for "duplicate session" — it falls through to the catch-all CommandFailed variant. This means:
- API layers can't distinguish "duplicate" from other failures
- HTTP returns 500 instead of 409 Conflict
- gRPC returns INTERNAL instead of ALREADY_EXISTS
- Error messages expose raw tmux stderr instead of a clean domain message

## Proposed change
1. Add `SessionAlreadyExists(String)` variant to TmuxError
2. Add detection in from_stderr() for "duplicate session" patterns
3. API layers can map this to appropriate transport responses (409/ALREADY_EXISTS)

Note: The transport mapping itself is an API concern (per #35), but the error variant belongs in the domain since "duplicate session" is a domain concept.

## Acceptance criteria
- [ ] TmuxError::SessionAlreadyExists(String) variant added
- [ ] from_stderr() detects "duplicate session" stderr patterns
- [ ] Unit tests for the new error classification
- [ ] Error message is clean: "session already exists: {name}"
