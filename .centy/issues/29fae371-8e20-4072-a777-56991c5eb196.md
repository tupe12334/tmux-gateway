# Log or error on unparseable tmux output lines instead of silently dropping them

list_sessions, list_windows, and list_panes use filter_map to parse tmux output. Lines that fail to parse (wrong number of tab-separated fields) are silently discarded with no logging or error.

## Problem
- sessions.rs line 38-42: filter_map returns None for lines with < 4 parts — silently dropped
- list_windows.rs line 38-42: same pattern
- list_panes.rs line 38-42: same pattern
- If tmux changes its output format or a field is missing, the domain silently returns fewer results than tmux reported
- Debugging missing sessions/windows/panes is extremely difficult when lines are dropped without trace

## What to do
- Replace silent filter_map(None) with explicit handling:
  - Option A: Log a warning for each unparseable line with the raw content (best for debugging)
  - Option B: Return a parse error if any line fails (strictest)
  - Option C: Collect parse warnings alongside results: Result<(Vec<T>, Vec<ParseWarning>), TmuxError>
- At minimum, add tracing::warn! for dropped lines so operators can detect format mismatches

## Acceptance criteria
- No tmux output line is silently discarded
- Unparseable lines produce a log entry or structured warning
- Operators can detect when tmux output format doesn't match expectations
