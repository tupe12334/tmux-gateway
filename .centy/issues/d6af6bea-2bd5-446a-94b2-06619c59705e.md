# Add CapturePaneResult value object with structured metadata

The \`capture_pane\` domain operation returns a raw \`String\`, losing useful metadata about the captured content. In DDD, the result should be a value object that carries structured information alongside the content.

## Problem
- \`capture_pane\` returns \`Result<String, TmuxError>\` — just raw text
- Callers cannot distinguish between an empty pane and a pane with only whitespace
- No metadata about the capture: line count, whether content was truncated, visible dimensions
- Issue #51 proposes capture options (scroll history, line ranges) which will make structured results even more important
- Issue #75 proposes normalization — the normalized vs raw distinction should be explicit in the return type

## Proposed solution
Introduce a \`CapturePaneResult\` value object:

\`\`\`rust
pub struct CapturePaneResult {
    /// The raw captured content.
    pub content: String,
    /// Number of lines in the captured content.
    pub line_count: usize,
    /// Whether trailing empty lines were present (before any normalization).
    pub has_trailing_whitespace: bool,
    /// The target pane that was captured.
    pub target: String,
}
\`\`\`

Update \`capture_pane\` to return \`Result<CapturePaneResult, TmuxError>\`.

This is a pure domain change — API layers decide how to serialize the result (REST might return the struct as JSON, GraphQL as a type with fields, gRPC as a message).

## Scope
- New \`CapturePaneResult\` struct in domain layer
- Update \`capture_pane\` return type
- Update \`TmuxCommands\` trait signature
- All API layers updated to handle the new return type
- Coordinates with #51 (capture options) and #75 (normalization)
