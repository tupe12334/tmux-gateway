# Add Timeout error variant and domain-level operation cancellation

TmuxError has no Timeout variant. Issue #13 adds timeouts at the infrastructure level, but there is no domain-level representation of a timed-out operation. Callers cannot distinguish "tmux command failed" from "tmux command took too long."

## Problem
- TmuxError has CommandFailed but no Timeout — a timeout surfaces as a generic CommandFailed or task join error
- No way for API layers to return a 408/504 vs 500 based on error type
- Issue #13 adds timeouts for spawn_blocking but wraps the timeout as a String error inside CommandFailed
- The domain should model timeout as a first-class concept

## What to do
Add a new TmuxError variant:

TmuxError::OperationTimedOut { command: String, timeout_ms: u64 }

When a tmux operation exceeds the configured timeout, return this variant instead of CommandFailed. This allows API layers to map it appropriately (HTTP 504, gRPC DEADLINE_EXCEEDED, GraphQL timeout extension).

## Acceptance criteria
- Timeout is a distinct error variant in TmuxError
- Timed-out operations are distinguishable from other failures at the type level
- API layers can map timeout errors to protocol-appropriate status codes
