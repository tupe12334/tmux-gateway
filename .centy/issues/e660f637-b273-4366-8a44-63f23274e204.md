# Align TmuxCommands trait with all public domain operations

## Problem

The `TmuxCommands` trait in `commands.rs` is the contract that forces all API layers (REST, GraphQL, gRPC) to implement every domain operation. However, it's incomplete:

**Currently in trait (16 methods):**
`ls`, `create_session`, `kill_session`, `kill_window`, `kill_pane`, `list_windows`, `list_panes`, `send_keys`, `rename_session`, `rename_window`, `new_window`, `split_window`, `capture_pane`, `create_session_with_windows`, `swap_panes`, `move_window`

**Public operations NOT in trait:**
- `server_info()` and `is_available()` from `server_info.rs` â€” exposed as free functions, not trait methods
- `get_session()` and `session_exists()` from `sessions.rs` â€” public but not in trait

When new operations are added (as proposed by 90+ open issues), there's no enforcement that API layers implement them. An operation could exist in the domain but be unreachable from one or more APIs.

## Proposed change

1. Add missing operations to the `TmuxCommands` trait:
   ```rust
   fn server_info(&self) -> impl Future<Output = Result<TmuxServerInfo, TmuxError>> + Send;
   fn is_available(&self) -> impl Future<Output = Result<bool, TmuxError>> + Send;
   fn get_session(&self, name: &str) -> impl Future<Output = Result<Option<TmuxSession>, TmuxError>> + Send;
   fn session_exists(&self, name: &str) -> impl Future<Output = Result<bool, TmuxError>> + Send;
   ```

2. Add a compile-time enforcement pattern: document that ALL new domain operations MUST be added to the trait before being merged. Consider a test that uses reflection/inventory to verify all public async functions are in the trait.

3. Group trait methods by category with doc comments:
   ```rust
   // --- Queries ---
   // --- Session Commands ---
   // --- Window Commands ---
   // --- Pane Commands ---
   // --- Composite Operations ---
   // --- Health ---
   ```

## Why this matters

- The trait is the single source of truth for API completeness
- Without trait enforcement, REST could have `server_info` but GraphQL might not
- As the issue backlog grows (90+ new operations proposed), this becomes critical
- Compile-time enforcement is the strongest guarantee â€” no runtime surprises
- API-agnostic by definition: the trait IS the API-agnostic interface

## Acceptance criteria

- [ ] `server_info`, `is_available`, `get_session`, `session_exists` added to trait
- [ ] All three API layers implement the new trait methods
- [ ] Trait methods grouped by category with doc comments
- [ ] All existing public operations in core crate are in the trait (verify exhaustively)
- [ ] Documentation: "new operations MUST be added to TmuxCommands trait"
