# Add target validation to capture_pane domain operation

capture_pane is the only pane-targeting domain operation that does not validate its target parameter. Every other pane operation (kill_pane, send_keys, split_window) calls validate_pane_target() before executing, but capture_pane passes the raw string directly to tmux.

## Problem

- `capture_pane(target: &str)` in `crates/tmux-gateway-core/src/capture_pane.rs` does not call any validation
- All other pane operations validate: kill_pane, send_keys, split_window
- This inconsistency means capture_pane could receive malformed targets that other operations would reject
- The gap exists across all API layers (REST, gRPC, GraphQL) since they all delegate to the same domain function

## Solution

Add `validate_pane_target(target)?` at the start of capture_pane(), matching the pattern used in kill_pane, send_keys, and split_window:

```rust
pub async fn capture_pane(target: &str) -> Result<String, TmuxError> {
    validation::validate_pane_target(target)?;  // Add this line
    // ... rest unchanged
}
```

## Impact

- Consistent validation across all pane operations
- Prevents malformed targets from reaching tmux
- All three API layers benefit automatically since they delegate to the domain function
