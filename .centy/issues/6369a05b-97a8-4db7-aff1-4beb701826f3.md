# Add send_keys input sanitization to prevent control sequence injection

send_keys accepts arbitrary strings and passes them directly to tmux send-keys. There is no validation or sanitization of the key values, allowing callers to inject arbitrary terminal escape sequences or control characters.

## Problem
- send_keys takes &[String] with no validation on individual key values
- A caller could send escape sequences that manipulate the terminal (clear screen, modify terminal state)
- tmux send-keys interprets certain strings specially (C-a, M-x, Enter, etc.) — no documentation or validation of which key names are valid
- Unlike other operations, there is no validate_* call before execution

## What to do
- Define a domain-level key validation function that:
  - Rejects null bytes and raw escape sequences (\\x1b, \\x00)
  - Optionally restricts to known tmux key names + printable characters
  - Enforces a maximum key count per request (prevent flooding)
  - Enforces a maximum key string length
- Add validation to send_keys before executing the tmux command
- Document which key formats are supported (literal text vs tmux key names like Enter, C-c, etc.)

## Acceptance criteria
- send_keys rejects inputs containing null bytes or raw escape sequences
- A maximum key count per request is enforced at the domain level
- The validation is called before the tmux command executes
