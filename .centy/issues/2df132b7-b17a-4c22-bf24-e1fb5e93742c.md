# Implement From trait for domain-to-API type conversions to eliminate duplication

All three API layers contain identical field-by-field mapping code to convert domain types (TmuxSession, TmuxWindow, TmuxPane) to their transport-specific types. This mapping is duplicated across REST, GraphQL, and gRPC layers.

## Problem
- REST maps TmuxSession -> SessionResponse with identical field assignments
- GraphQL maps TmuxSession -> Session with identical field assignments
- gRPC maps TmuxSession -> proto::TmuxSession with identical field assignments
- Same pattern repeated for TmuxWindow and TmuxPane
- If a field is added to a domain type, three files must be updated

## What to do
- Implement From<domain::TmuxSession> for each API-layer type:
  - impl From<TmuxSession> for rest::SessionResponse
  - impl From<TmuxSession> for graphql::Session
  - impl From<TmuxSession> for proto::TmuxSession
- Same for TmuxWindow and TmuxPane conversions
- API handler code simplifies to: sessions.into_iter().map(Into::into).collect()

## Acceptance criteria
- No manual field-by-field mapping in handler functions
- Each API type has a From impl that owns its conversion logic
- Adding a field to a domain type produces a compiler error in each From impl (no silent omission)
