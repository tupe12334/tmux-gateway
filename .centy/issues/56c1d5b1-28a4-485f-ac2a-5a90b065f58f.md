# Add domain error recovery classification for operation retry guidance

## Problem

`TmuxError` variants represent different failure modes, but the domain provides no guidance on which errors are transient (worth retrying) vs. permanent (will always fail). Callers must make their own retry decisions, leading to inconsistent behavior:

- `TmuxNotRunning` might be transient (server starting up) or permanent (tmux not installed)
- `CommandFailed` could be a momentary lock contention or a fundamental misconfiguration
- `SessionNotFound` is almost always permanent for a given target
- `Validation` errors are always permanent — retrying is wasteful

Without this classification, API layers either retry everything (wasteful, slow) or nothing (fragile).

## Proposed change

Add error classification methods to `TmuxError`:

```rust
#[derive(Debug, Clone, Copy, PartialEq, Eq)]
pub enum ErrorRecoverability {
    /// Error will not resolve by retrying with the same input.
    Permanent,
    /// Error may resolve if retried after a short delay.
    Transient,
    /// Recoverability depends on external state — caller should inspect context.
    Unknown,
}

impl TmuxError {
    /// Classify this error's recoverability to guide retry decisions.
    pub fn recoverability(&self) -> ErrorRecoverability {
        match self {
            Self::SessionNotFound(_) | Self::WindowNotFound(_) | Self::PaneNotFound(_) => {
                ErrorRecoverability::Permanent
            }
            Self::SessionAlreadyExists(_) => ErrorRecoverability::Permanent,
            Self::InvalidTarget(_) => ErrorRecoverability::Permanent,
            Self::Validation(_) => ErrorRecoverability::Permanent,
            Self::ParseError { .. } => ErrorRecoverability::Permanent,
            Self::TmuxNotRunning => ErrorRecoverability::Transient,
            Self::CommandFailed { .. } => ErrorRecoverability::Unknown,
        }
    }

    /// Whether this error is safe to retry without side effects.
    pub fn is_retryable(&self) -> bool {
        matches!(
            self.recoverability(),
            ErrorRecoverability::Transient | ErrorRecoverability::Unknown
        )
    }
}
```

This is pure domain logic — no retry mechanism, no timers, no infrastructure. Just classification that callers (API layers, operation composers) can use to make informed decisions.

## Why this matters

- Enables intelligent retry behavior in API middleware without per-error-variant switch statements
- Makes error handling consistent across REST, gRPC, and GraphQL layers
- Informs gRPC status codes: transient errors → `UNAVAILABLE` (retry), permanent → `NOT_FOUND`/`INVALID_ARGUMENT` (don't retry)
- Complements #142 (operation composition with rollback) — rollback decisions depend on error recoverability
- Complements #151 (DomainConfig) — retry policies reference this classification
- Zero overhead: simple pattern match, no allocations
