# Add break-pane and join-pane domain operations for pane-window reorganization

Issue #71 covers swap and move operations, but two fundamental pane reorganization commands are missing: break-pane (promote a pane to its own window) and join-pane (merge a pane from another window into the current one). These are structural operations that change the session hierarchy.

## Problem
- No way to promote a pane into its own window through the gateway
- No way to merge a pane from one window into another
- These are among the most-used tmux commands for workspace reorganization
- Together with split-window (#76) and swap (#71), they complete the pane lifecycle

## Proposed solution

\`\`\`rust
/// Promote a pane to its own new window within the same session.
/// Returns the new window's identifier.
pub async fn break_pane(target: &str) -> Result<String, TmuxError>

/// Move a source pane into the target window, creating a new split.
/// \`horizontal\` controls split direction (like split-window).
pub async fn join_pane(
    source: &str,   // pane target to move (e.g. "session:1.2")
    target: &str,   // destination pane target (e.g. "session:0.0")
    horizontal: bool,
) -> Result<(), TmuxError>
\`\`\`

Both operations:
- Validate targets via existing validation functions
- Use tmux \`break-pane -t\` and \`join-pane -s source -t target\` respectively
- Map errors through \`TmuxError::from_stderr\`

## Scope
- New \`break_pane.rs\` and \`join_pane.rs\` modules in \`tmux-gateway-core\`
- Add both to \`TmuxCommands\` trait
- Complements #71 (swap/move) to complete pane reorganization operations
