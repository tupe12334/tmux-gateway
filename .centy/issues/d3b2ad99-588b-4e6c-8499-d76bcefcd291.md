# Introduce a TmuxExecutor port trait to decouple domain from tmux_interface crate

The domain operations in tmux-gateway-core directly depend on the tmux_interface crate, constructing tmux_interface command objects and calling .output() inside spawn_blocking. This tight coupling prevents unit testing domain logic without a real tmux binary and violates the ports-and-adapters pattern.

## Problem

- Every operation file imports and uses tmux_interface types directly (Tmux, ListSessions, SendKeys, etc.)
- Domain operations cannot be tested without a running tmux server
- Swapping the tmux backend (e.g., for a mock, a remote tmux over SSH, or a different terminal multiplexer) requires changing every operation file
- The core crate's Cargo.toml has tmux_interface as a hard dependency
- Issue #40 (Separate tmux I/O adapter) and #49 (Stop re-exporting tmux_interface) are related but don't define the port trait

## Solution

Define a port trait in the domain layer that abstracts tmux command execution:

```rust
// Domain port (in tmux-gateway-core)
pub trait TmuxExecutor: Send + Sync {
    fn execute(&self, command: &str, args: &[&str]) -> Result<CommandOutput, TmuxError>;
    fn execute_with_format(&self, command: &str, args: &[&str], format: &str) -> Result<String, TmuxError>;
}

pub struct CommandOutput {
    pub stdout: String,
    pub stderr: String,
    pub success: bool,
}
```

Then create the real adapter in a separate module/crate:

```rust
// Infrastructure adapter
pub struct LocalTmuxExecutor;
impl TmuxExecutor for LocalTmuxExecutor {
    fn execute(&self, command: &str, args: &[&str]) -> Result<CommandOutput, TmuxError> {
        // Uses tmux_interface here
    }
}
```

Domain operations receive the executor as a parameter or via a context:

```rust
pub fn list_sessions(executor: &dyn TmuxExecutor) -> Result<Vec<TmuxSession>, TmuxError> {
    let output = executor.execute_with_format("list-sessions", &[], FORMAT_STRING)?;
    parse_sessions_output(&output)
}
```

## Impact

- Domain operations become testable with a mock executor (no tmux needed)
- Clear ports-and-adapters architecture
- Enables alternative backends (remote tmux, test doubles, recording/replay)
- tmux_interface becomes an implementation detail of the adapter, not a domain dependency
