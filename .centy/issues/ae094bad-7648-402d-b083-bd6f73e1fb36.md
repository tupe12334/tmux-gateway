# Replace boolean parameters with domain enums across operations

## Problem

Several domain operations use boolean parameters that obscure intent:

1. `split_window(target, horizontal: bool)` — callers must remember what `true` means
2. `TmuxSession.attached: bool` — doesn't distinguish 1 client from 5 clients
3. `TmuxWindow.active: bool` / `TmuxPane.active: bool` — boolean flag when an enum is clearer

Boolean parameters violate DDD principles: they carry no domain meaning and force callers to consult documentation to understand what `true` vs `false` represents.

## Proposed change

Add domain enums in `tmux-gateway-core`:

```rust
#[derive(Clone, Copy, Debug, PartialEq, Eq)]
pub enum SplitDirection {
    Horizontal,
    Vertical,
}

#[derive(Clone, Copy, Debug, PartialEq, Eq)]
pub enum AttachState {
    Attached { client_count: u32 },
    Detached,
}

#[derive(Clone, Copy, Debug, PartialEq, Eq)]
pub enum ActiveState {
    Active,
    Inactive,
}
```

Update signatures:
- `split_window(target, direction: SplitDirection)` instead of `horizontal: bool`
- `TmuxSession { attached: AttachState }` instead of `bool`
- `TmuxWindow { active: ActiveState }` / `TmuxPane { active: ActiveState }`

## Why this matters

- Self-documenting code: `SplitDirection::Horizontal` vs `true`
- Extensible: `AttachState::Attached { client_count }` carries more info than a boolean
- API layers can expose enums directly (GraphQL enum, protobuf enum)
- Pattern matching is exhaustive — no forgotten branches
- Aligns with DDD principle of making implicit concepts explicit

## Acceptance criteria

- [ ] `SplitDirection`, `AttachState`, `ActiveState` enums in core crate
- [ ] `split_window` signature updated to use `SplitDirection`
- [ ] `TmuxSession.attached` changed to `AttachState`
- [ ] `TmuxWindow.active` and `TmuxPane.active` changed to `ActiveState`
- [ ] Parsing updated to construct enums from tmux output
- [ ] All API layers updated (non-breaking: accept both enum and legacy boolean in transition)
- [ ] Unit tests for enum construction from tmux output values
