# Add domain-level authorization model for operation access control

Issue #18 covers API key authentication at the transport layer, but the domain has no concept of permissions or authorization. In production, different clients should have different access levels (read-only monitoring vs full control). Authorization is a domain concern, not a transport concern.

## Problem

- Any authenticated client can execute any operation (kill-server, send-keys, etc.)
- No domain concept of read vs write vs admin operations
- Multi-tenant deployments have no way to restrict session access per client
- The TmuxCommands trait treats all operations equally with no permission checks
- Transport-level auth (#18) only answers "who are you?" not "what can you do?"

## Solution

Add a domain authorization model that classifies operations and checks permissions:

```rust
pub enum OperationClass {
    Read,       // ls, list-windows, list-panes, capture-pane, has-session
    Write,      // new-session, send-keys, rename, split-window
    Destructive, // kill-session, kill-window, kill-pane, kill-server
    Admin,      // server config, hooks, source-file
}

pub trait AuthorizationPolicy: Send + Sync {
    fn authorize(&self, principal: &str, operation: OperationClass, target: &str) -> Result<(), AuthorizationError>;
}

#[derive(Debug, thiserror::Error)]
pub enum AuthorizationError {
    #[error("operation {operation:?} denied for principal {principal}")]
    Denied { principal: String, operation: OperationClass },
    #[error("principal {principal} has no access to target {target}")]
    TargetRestricted { principal: String, target: String },
}
```

Domain operations accept an authorization context:
```rust
pub async fn kill_session(
    auth: &dyn AuthorizationPolicy,
    principal: &str,
    target: &str,
) -> Result<(), TmuxError> {
    auth.authorize(principal, OperationClass::Destructive, target)?;
    // ... proceed
}
```

## Impact

- Domain-level access control independent of transport
- Fine-grained per-operation and per-target authorization
- Enables multi-tenant deployments with isolated session access
- API layers pass through the authorization context, keeping the policy in the domain
