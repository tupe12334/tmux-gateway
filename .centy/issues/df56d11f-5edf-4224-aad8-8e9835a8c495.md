# Add domain-level aggregate metrics for operational monitoring

The domain has per-request observability (tracing spans #62, audit journal #127) but no aggregate metrics. Production deployments need counters and gauges for monitoring dashboards: active sessions, operations per minute, error rates by type, average operation latency.

## Problem

- No domain-level concept of operational metrics or counters
- Monitoring dashboards must derive metrics from logs or external instrumentation
- No way to answer "how many sessions are active?" without calling list_sessions
- Error rates by type (SessionNotFound vs CommandFailed vs timeout) are invisible
- Operation throughput and latency distribution are unknown at the domain level
- Issue #62 (tracing spans) covers per-request telemetry, not aggregate counters
- Issue #127 (audit journal) logs events but doesn't aggregate them

## Solution

Add a domain metrics model that operations report into:

```rust
pub struct DomainMetrics {
    pub operations_total: HashMap<&'static str, u64>,
    pub operations_failed: HashMap<&'static str, u64>,
    pub errors_by_type: HashMap<String, u64>,
    pub active_sessions: u32,
    pub total_sessions_created: u64,
    pub total_sessions_destroyed: u64,
}

pub trait MetricsCollector: Send + Sync {
    fn record_operation(&self, name: &'static str, duration: Duration, result: OperationOutcome);
    fn snapshot(&self) -> DomainMetrics;
}

pub enum OperationOutcome {
    Success,
    Failed(String), // error type name
}
```

Domain operations report metrics through the collector:

```rust
pub async fn new_session(metrics: &dyn MetricsCollector, name: &str) -> Result<String, TmuxError> {
    let start = Instant::now();
    let result = /* execute */;
    metrics.record_operation("new_session", start.elapsed(), result.as_outcome());
    result
}
```

## Impact

- Enables production monitoring dashboards without log parsing
- Domain-level metrics are API-agnostic — available to all transport layers
- Supports alerting on error rate spikes or session count thresholds
- Complements tracing (#62) and audit (#127) for full observability stack
