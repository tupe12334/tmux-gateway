# Add session lifecycle state machine with enforced transitions

## Problem

Tmux sessions have implicit lifecycle states (active, detached, zombie/dead) but the domain layer treats all sessions identically via a flat `TmuxSession` struct with a boolean `attached` field. There is no concept of valid state transitions, and operations that are invalid in certain states (e.g., sending keys to a dead session) are only caught by tmux stderr parsing after the fact.

## Proposed change

Add a `SessionState` enum and transition logic as pure domain types:

```rust
#[derive(Clone, Debug, PartialEq, Eq)]
pub enum SessionState {
    Active { client_count: u32 },
    Detached,
    // Future: Zombie (session exists but all windows dead)
}

impl SessionState {
    /// Pure: derive state from tmux session attributes
    pub fn from_session(session: &TmuxSession) -> Self;
}

/// Pure: check if an operation is valid for the current session state
pub fn validate_operation_for_state(
    state: &SessionState,
    operation: &OperationType,
) -> Result<(), TmuxError>;
```

Valid transitions (enforced as pure functions):

- `Active → Detached` (all clients disconnect)
- `Detached → Active` (client attaches)
- `Active|Detached → Killed` (kill-session)

Operations gated by state:

- `send_keys`: requires `Active` or `Detached` (session must exist)
- `capture_pane`: valid in any non-killed state
- `rename_session`: valid in any non-killed state
- `kill_session`: valid in any state (idempotent)

## Why this matters

- Gives callers fast, domain-level feedback instead of waiting for tmux to fail
- Issue #85 (precondition checks) needs state awareness to validate preconditions
- Issue #67 (idempotency) can use state checks: `create_session` on Active → noop
- Issue #109 (domain invariants) needs formal state rules to enforce

## Acceptance criteria

- [ ] `SessionState` enum in `tmux-gateway-core`
- [ ] Pure `from_session()` constructor from `TmuxSession` attributes
- [ ] Pure `validate_operation_for_state()` function
- [ ] Unit tests for all state derivation cases
- [ ] Unit tests for valid/invalid operation-state combinations
- [ ] No I/O — state is derived from already-fetched domain objects
