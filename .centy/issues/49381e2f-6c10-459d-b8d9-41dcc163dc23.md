# Replace fragile stderr string matching in error classification with structured parsing

TmuxError::from_stderr() classifies errors by checking if the stderr string contains substrings like "session not found", "can't find session", "no server running". This approach is fragile and locale-dependent.

## Problem

- `from_stderr()` in `crates/tmux-gateway-core/src/error.rs` uses `.to_lowercase().contains()` on stderr text
- tmux error messages can change between versions (e.g., "can't find session" vs "session not found")
- If tmux is compiled with different locales, messages may differ
- The string patterns are hardcoded and not tested against real tmux version variations
- Any unrecognized error falls through to generic CommandFailed, losing semantic meaning

## Solution

1. Use tmux exit codes as the primary classification signal (non-zero = error)
2. Parse the structured parts of stderr (e.g., the target name that failed) rather than matching English phrases
3. Add a pattern registry that maps known stderr patterns to error variants, making it easy to add new patterns
4. Add a fallback that preserves the raw stderr for unknown errors
5. Consider using `tmux -f /dev/null` format flags where available to get machine-readable output

## Impact

- More reliable error classification across tmux versions and locales
- Domain error types accurately reflect what went wrong
- All API layers get better error responses automatically
- Easier to add error patterns for new tmux versions
