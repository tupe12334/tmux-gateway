# Add domain-level resize operations for panes and windows

## Problem

Resizing panes and windows is a fundamental tmux operation (`resize-pane`, `resize-window`) that the gateway currently does not expose. Users who manage complex layouts through the gateway cannot adjust pane proportions, and issue #69 (layout management) lists layouts as templates but doesn't cover the granular resize operations that layouts are built from.

The existing `PaneGeometry` concept from `list_panes` provides read access to dimensions, but there's no write counterpart.

## Proposed change

Add resize domain operations:

```rust
pub enum ResizeDirection {
    Up,
    Down,
    Left,
    Right,
}

pub enum ResizeAmount {
    /// Resize by N cells in the given direction
    Cells(u32),
    /// Resize to an absolute percentage of the window
    Percentage(u32),
    /// Resize to exact cell dimensions
    Absolute { width: u32, height: u32 },
}

/// Resize a pane in the given direction by the specified amount.
pub async fn resize_pane(
    executor: &impl TmuxExecutor,
    target: &str,
    direction: ResizeDirection,
    amount: ResizeAmount,
) -> Result<(), TmuxError>;

/// Resize a window to absolute dimensions.
pub async fn resize_window(
    executor: &impl TmuxExecutor,
    target: &str,
    width: u32,
    height: u32,
) -> Result<(), TmuxError>;

/// Apply a named layout to a window (even-horizontal, even-vertical, main-horizontal, main-vertical, tiled).
pub async fn apply_layout(
    executor: &impl TmuxExecutor,
    target: &str,
    layout: &str,
) -> Result<(), TmuxError>;
```

Value objects `ResizeDirection` and `ResizeAmount` encapsulate the resize semantics in a type-safe, API-agnostic way.

## Why this matters

- Completes the pane management surface: create, select, split, swap, zoom (#158), capture — but no resize
- Essential for building and adjusting layouts programmatically
- `ResizeAmount` value object prevents invalid inputs (negative sizes, >100% percentages) at the domain level
- Enables automation: "split window, then resize left pane to 30%"
- Prerequisite for layout management (#69) — layouts are sequences of splits and resizes
- API-agnostic: all three APIs expose the same domain operations

## Acceptance criteria

- [ ] `ResizeDirection` enum (Up, Down, Left, Right)
- [ ] `ResizeAmount` enum (Cells, Percentage, Absolute)
- [ ] `resize_pane(executor, target, direction, amount)` maps to `tmux resize-pane` with appropriate flags
- [ ] `resize_window(executor, target, width, height)` maps to `tmux resize-window`
- [ ] `apply_layout(executor, target, layout)` maps to `tmux select-layout`
- [ ] Validation: percentage 1-100, cells > 0, absolute dimensions > 0
- [ ] Target validation before execution
- [ ] Unit tests with mock executor for each resize variant
- [ ] Error mapping for invalid targets and impossible resizes (e.g., pane already minimum size)
