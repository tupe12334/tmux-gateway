# Add property-based fuzz testing for domain parse functions

## Problem

Parse functions (`parse_session_line`, `parse_window_line`, `parse_pane_line`) are tested with hand-written cases covering expected patterns and a few error cases. However, they haven't been tested against:

- Overflow values (u32::MAX, i64::MAX for timestamps)
- Tab characters embedded within fields (format string confusion)
- Fields with only whitespace
- Empty tab-separated fields (`"\t\t\t"`)
- Very long field values (megabyte-length session names)
- Unicode edge cases (emoji, zero-width characters, RTL overrides)
- Null bytes embedded in strings

These edge cases can cause silent misparsing, panics, or excessive memory allocation.

## Proposed change

Add property-based tests using the `proptest` crate (dev dependency only) in `tmux-gateway-core`:

```rust
#[cfg(test)]
mod prop_tests {
    use proptest::prelude::*;

    proptest! {
        #[test]
        fn parse_session_line_never_panics(line in ".*") {
            let _ = parse_session_line(&line);
        }

        #[test]
        fn parse_session_line_roundtrip(
            name in "[a-zA-Z0-9_.-]{1,128}",
            windows in 0u32..10000,
            created in 0i64..2000000000,
            attached in prop::bool::ANY,
        ) {
            let line = format!("{name}\t{windows}\t{created}\t{}", if attached { "1" } else { "0" });
            let session = parse_session_line(&line).unwrap();
            assert_eq!(session.name, name);
            assert_eq!(session.windows, windows);
            assert_eq!(session.created, created);
            assert_eq!(session.attached, attached);
        }
    }
}
```

Similar property tests for `parse_window_line` and `parse_pane_line`.

Key properties to verify:
1. **No panics**: arbitrary input never causes a panic (returns Err instead)
2. **Round-trip**: well-formed input parses correctly and can be reconstructed
3. **Bounds**: parsed numeric values are within expected ranges
4. **Error quality**: malformed input produces ParseError (not CommandFailed or other variants)

## Why this matters

- Parse functions are the boundary between tmux (untrusted external output) and domain types
- A malformed tmux output should never crash the server
- Issue #22 fixed some unwrap_or(0) silent failures; property testing prevents regressions
- All tests are pure — no tmux dependency, run in milliseconds
- Catches edge cases that hand-written tests miss

## Acceptance criteria

- [ ] `proptest` added as dev dependency to `tmux-gateway-core`
- [ ] Property tests for `parse_session_line`, `parse_window_line`, `parse_pane_line`
- [ ] "Never panics" property verified for arbitrary string input
- [ ] "Round-trip" property verified for well-formed input
- [ ] "Correct error variant" property: malformed input → ParseError
- [ ] At least 1000 random inputs per property (proptest default)
