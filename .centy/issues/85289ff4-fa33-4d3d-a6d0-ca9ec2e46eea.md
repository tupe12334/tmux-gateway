# Add ANSI-aware pane capture with terminal formatting preservation

## Problem

The existing `capture_pane` and `capture_pane_with_options` operations strip terminal formatting by default. While `CaptureOptions.escape_sequences` exists as a boolean flag, there is no domain-level abstraction for working with ANSI-formatted pane content — the raw escape codes are opaque to clients.

Clients building terminal UIs, web-based terminal renderers, or log viewers need:
1. Structured representation of colored/styled terminal output
2. The ability to capture with escape codes AND have the domain layer help interpret them
3. Conversion utilities (ANSI to HTML, ANSI to plain text with style metadata)

## Proposed change

Add ANSI-aware capture and conversion utilities in the domain layer:

```rust
/// Represents a styled span of terminal text.
#[derive(Debug, Clone, PartialEq, Eq)]
pub struct StyledSpan {
    pub text: String,
    pub foreground: Option<TerminalColor>,
    pub background: Option<TerminalColor>,
    pub bold: bool,
    pub underline: bool,
    pub italic: bool,
}

#[derive(Debug, Clone, PartialEq, Eq)]
pub enum TerminalColor {
    Named(u8),       // 0-15 standard colors
    Palette(u8),     // 0-255 palette
    Rgb(u8, u8, u8), // 24-bit true color
}

/// Capture pane content with ANSI escape codes and parse into styled spans.
pub async fn capture_pane_styled(
    executor: &impl TmuxExecutor,
    target: &str,
    opts: &CaptureOptions,
) -> Result<Vec<Vec<StyledSpan>>, TmuxError>;

/// Convert styled spans to HTML with inline styles.
pub fn styled_to_html(lines: &[Vec<StyledSpan>]) -> String;

/// Strip all styling, returning plain text.
pub fn styled_to_plain(lines: &[Vec<StyledSpan>]) -> String;
```

The ANSI parsing is pure domain logic — no I/O, fully testable with static input strings.

## Why this matters

- Enables rich terminal rendering in web UIs and dashboards
- ANSI parsing is pure, stateless logic — ideal for the functional core
- `StyledSpan` and `TerminalColor` are value objects that fit the DDD model
- Complements `capture_pane_with_options` (escape_sequences=true) with structured output
- API-agnostic: REST returns HTML, GraphQL returns typed spans, gRPC returns structured messages
- No new I/O dependencies — pure string parsing in the domain layer

## Acceptance criteria

- [ ] `StyledSpan` value object with text, colors, and style flags
- [ ] `TerminalColor` enum supporting named, palette, and RGB colors
- [ ] `capture_pane_styled` combines capture with ANSI parsing
- [ ] `styled_to_html` converts styled spans to HTML with inline styles
- [ ] `styled_to_plain` strips all styling
- [ ] Handles SGR reset sequences correctly
- [ ] Unit tests with known ANSI escape sequences
- [ ] Handles edge cases: incomplete sequences, nested styles, 256-color, true color
- [ ] No external ANSI parsing dependency (or minimal, well-vetted crate)
