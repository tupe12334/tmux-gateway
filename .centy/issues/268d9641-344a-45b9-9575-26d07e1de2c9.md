# Add display-message domain operation for dynamic tmux variable queries

tmux \`display-message -p\` evaluates format strings and returns the result, enabling queries for any tmux variable (#{cursor_x}, #{pane_pid}, #{window_layout}, etc.) without adding dedicated operations for each one.

## Problem
- Each new piece of tmux state requires a new domain operation (e.g. #55 enriches TmuxPane, #58 adds IDs)
- tmux has 100+ format variables that could be useful to callers
- No generic way to query arbitrary tmux state through the gateway
- Adding operations one-by-one doesn't scale; a generic query mechanism is needed

## Proposed solution

\`\`\`rust
/// Evaluate a tmux format string against a target and return the result.
pub async fn display_message(
    target: &str,          // session, window, or pane target
    format_string: &str,   // tmux format string, e.g. "#{pane_pid}"
) -> Result<String, TmuxError>
\`\`\`

Validation:
- Target validated via existing validation functions
- Format string validated: must contain only safe characters, must include at least one \`#{\` variable reference
- Reject format strings containing shell escape sequences or command substitution

This provides an extensible escape hatch: callers can query any tmux variable without waiting for dedicated domain operations, while the domain still enforces safety.

## Scope
- New \`display_message.rs\` module in \`tmux-gateway-core\`
- Format string validation in \`validation.rs\`
- Add to \`TmuxCommands\` trait
- Provides generic alternative to per-variable domain model enrichment
