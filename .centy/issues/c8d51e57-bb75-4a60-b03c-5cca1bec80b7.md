# Add session snapshot and restore domain operations for state persistence

Issue #81 proposes a declarative session specification for template-based creation, but there is no way to capture the current state of an existing session as a specification and restore it later. This bidirectional capability enables session persistence, migration, and disaster recovery.

## Problem
- Sessions are lost when the tmux server restarts — no persistence
- No way to capture a running session's topology (windows, panes, working directories, layouts) as a portable description
- No way to migrate a session layout from one server to another
- #81 creates sessions from a spec, but there's no way to generate a spec from a running session

## Proposed solution

\`\`\`rust
/// A complete snapshot of a session's topology.
pub struct SessionSnapshot {
    pub name: String,
    pub windows: Vec<WindowSnapshot>,
}

pub struct WindowSnapshot {
    pub name: String,
    pub layout: String,              // tmux layout string
    pub panes: Vec<PaneSnapshot>,
    pub active: bool,
}

pub struct PaneSnapshot {
    pub current_path: String,        // working directory
    pub current_command: String,     // running process
    pub active: bool,
    pub width: u32,
    pub height: u32,
}

/// Capture the current state of a session as a snapshot.
pub async fn snapshot_session(target: &str) -> Result<SessionSnapshot, TmuxError>

/// Restore a session from a snapshot. Creates a new session with the same
/// topology, working directories, and layout.
pub async fn restore_session(
    snapshot: &SessionSnapshot,
    new_name: Option<&str>,  // None = use original name
) -> Result<String, TmuxError>
\`\`\`

This is a composite domain operation built on top of list-windows, list-panes, display-message (#98), and new-session/new-window/split-window.

## Scope
- New \`snapshot.rs\` module in \`tmux-gateway-core\`
- \`SessionSnapshot\`, \`WindowSnapshot\`, \`PaneSnapshot\` domain types
- \`snapshot_session\` and \`restore_session\` operations
- Add to \`TmuxCommands\` trait
- Builds on #81 (declarative spec), #74 (hierarchical model), #98 (display-message)
