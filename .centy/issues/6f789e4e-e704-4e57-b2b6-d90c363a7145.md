# Add structured error response format across all APIs

Error responses are plain strings across REST and GraphQL. Clients must parse error messages to determine the cause.

## Current State
- REST: Returns (StatusCode, String) tuple - no structured error body
- GraphQL: Converts TmuxError to string via .to_string()
- gRPC: Uses proper status codes but no structured error details

## Proposed Error Response
REST should return JSON error body:
{ "error": { "code": "SESSION_NOT_FOUND", "message": "session 'foo' not found", "request_id": "abc-123" } }

## Implementation
- Define ErrorResponse struct with utoipa schema
- Map TmuxError variants to error codes (SESSION_NOT_FOUND, INVALID_INPUT, etc.)
- Include request_id (from observability issue) when available
- Add error response schema to OpenAPI spec
- Use async-graphql error extensions for GraphQL
- Use tonic Status::with_details for gRPC
