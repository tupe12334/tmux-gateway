# Add max-length validation for target identifier strings

## Problem

The `validate_target_chars()` function in `crates/tmux-gateway-core/src/validation.rs` checks that target strings contain only safe characters but does not enforce a maximum length. A caller could pass a multi-megabyte target string that the validator would iterate character-by-character before eventually passing it to tmux.

`validate_session_name()` already has MAX_SESSION_NAME_LEN (128), but `validate_session_target()`, `validate_window_target()`, and `validate_pane_target()` have no length limit.

## Impact

- CPU waste iterating extremely long strings in validation
- Potential memory pressure when these strings are cloned for spawn_blocking closures
- tmux itself would reject them, but the rejection happens much later in the pipeline

## Proposed changes

- Add a MAX_TARGET_LEN constant (e.g., 256 — generous enough for `session:window.pane` format)
- Add length check at the top of `validate_session_target()`, `validate_window_target()`, and `validate_pane_target()`
- Return `ValidationError::InvalidTarget { reason }` for oversized targets

## Acceptance criteria

- [ ] All target validation functions reject inputs exceeding MAX_TARGET_LEN
- [ ] Length check happens before character iteration for early rejection
- [ ] Existing tests continue to pass
- [ ] New tests cover the boundary (exactly at limit, one over)
