# Add resource limit policies to prevent unbounded session/window/pane creation

The domain has no concept of resource limits. Any client can create unlimited sessions, windows, and panes, potentially exhausting system resources (memory, file descriptors, tmux server capacity). Resource limits are a domain invariant, not an API concern.

## Problem

- No limit on total sessions, windows per session, or panes per window
- A single misbehaving client can exhaust tmux server resources
- split_window can create unbounded panes in a single window
- No way to express organizational policies about resource usage
- tmux itself has no built-in limits — the gateway is the enforcement point

## Solution

Add domain-level resource limit policies:

```rust
pub struct ResourceLimits {
    pub max_sessions: Option<u32>,
    pub max_windows_per_session: Option<u32>,
    pub max_panes_per_window: Option<u32>,
    pub max_total_panes: Option<u32>,
}

impl Default for ResourceLimits {
    fn default() -> Self {
        Self {
            max_sessions: Some(50),
            max_windows_per_session: Some(20),
            max_panes_per_window: Some(16),
            max_total_panes: Some(200),
        }
    }
}

#[derive(Debug, thiserror::Error)]
pub enum ResourceLimitError {
    #[error("session limit reached ({current}/{max})")]
    SessionLimitReached { current: u32, max: u32 },
    #[error("window limit reached for session {session} ({current}/{max})")]
    WindowLimitReached { session: String, current: u32, max: u32 },
    #[error("pane limit reached for window {window} ({current}/{max})")]
    PaneLimitReached { window: String, current: u32, max: u32 },
}
```

Check limits before creation operations:
```rust
pub async fn new_session(limits: &ResourceLimits, name: &str) -> Result<String, TmuxError> {
    if let Some(max) = limits.max_sessions {
        let current = list_sessions().await?.len() as u32;
        if current >= max {
            return Err(TmuxError::ResourceLimitReached { ... });
        }
    }
    // ... proceed with creation
}
```

## Impact

- Prevents resource exhaustion from runaway or misbehaving clients
- Domain-level enforcement independent of transport or auth
- Configurable limits per deployment
- Clear error messages when limits are reached
