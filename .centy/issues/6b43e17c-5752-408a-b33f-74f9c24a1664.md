# Add domain-level operation timeout with configurable deadline

Domain operations delegate to tokio::task::spawn_blocking for tmux command execution but have no timeout. A hung tmux process will block the thread pool indefinitely. Issue #13 covers tmux operation timeout at the infrastructure level, but the domain layer should express timeout as a domain concept.

## Problem

- spawn_blocking tasks have no deadline — a hung tmux command blocks a thread forever
- No way for callers to specify operation urgency (e.g., health check should be fast, capture_pane can be slower)
- Thread pool exhaustion affects all operations when any single tmux call hangs
- The domain layer has no concept of operation deadline or time budget

## Solution

Add a domain-level timeout wrapper that all operations use:

```rust
// Domain concept: operation deadline
pub struct OperationTimeout(Duration);

impl Default for OperationTimeout {
    fn default() -> Self { Self(Duration::from_secs(10)) }
}

// In the tmux executor:
pub async fn execute_with_timeout<T>(
    timeout: OperationTimeout,
    task: impl FnOnce() -> Result<T, TmuxError> + Send + 'static,
) -> Result<T, TmuxError>
where T: Send + 'static {
    tokio::time::timeout(timeout.0, tokio::task::spawn_blocking(task))
        .await
        .map_err(|_| TmuxError::OperationTimedOut)?
        .map_err(|e| TmuxError::CommandFailed { ... })?
}
```

Add a new error variant:
```rust
pub enum TmuxError {
    // ... existing variants
    OperationTimedOut { command: String, timeout: Duration },
}
```

## Impact

- Prevents thread pool exhaustion from hung tmux processes
- Domain expresses timeout as a first-class concept
- API layers can map timeout errors to appropriate status codes (408/504/DEADLINE_EXCEEDED)
- Configurable per-operation for different use cases
