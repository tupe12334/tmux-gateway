# Add domain operation for pane scrollback history search

## Problem

Tmux panes accumulate scrollback history (often thousands of lines). Currently the gateway only supports `capture_pane` which retrieves raw content, forcing clients to download the entire buffer and search client-side. This is inefficient for common use cases like "find the last error message" or "locate a specific log line."

Tmux itself supports `copy-mode` search, but there is no programmatic search-in-scrollback API. The gateway can provide this as a domain-level operation by combining `capture_pane_with_options` with in-domain filtering.

## Proposed change

Add a `search_pane_history` domain operation:

```rust
pub struct PaneSearchOptions {
    /// Regex pattern to search for
    pub pattern: String,
    /// Maximum number of matches to return (0 = unlimited)
    pub max_matches: usize,
    /// Search direction: newest-first or oldest-first
    pub newest_first: bool,
    /// How many lines of scrollback to search (None = all available)
    pub history_limit: Option<i32>,
}

pub struct PaneSearchMatch {
    /// The line number in the scrollback buffer (negative = history)
    pub line_number: i32,
    /// The full text of the matching line
    pub line_text: String,
}

pub async fn search_pane_history(
    executor: &impl TmuxExecutor,
    target: &str,
    options: &PaneSearchOptions,
) -> Result<Vec<PaneSearchMatch>, TmuxError>
```

Implementation: uses `capture_pane_with_options` to retrieve the scrollback range, then applies regex matching in the domain layer. This keeps the heavy lifting in Rust (fast) while the tmux interaction remains a simple capture.

## Why this matters

- Eliminates round-trip waste: clients don't need to pull megabytes of scrollback to find one line
- Enables automation patterns: "wait until error appears in output", "find the last command prompt"
- Pure domain logic (regex matching) on top of existing infrastructure (`capture_pane_with_options`)
- Complements #146 (structured extraction) with a search-focused interface
- API-agnostic: all three APIs benefit equally

## Acceptance criteria

- [ ] `PaneSearchOptions` value object with pattern, max_matches, newest_first, history_limit
- [ ] `PaneSearchMatch` result type with line_number and line_text
- [ ] `search_pane_history` function using `capture_pane_with_options` + regex
- [ ] Pattern validation (reject invalid regex before executing tmux commands)
- [ ] Respects `max_matches` limit for bounded responses
- [ ] `newest_first` ordering supported
- [ ] Unit tests with mock executor covering: match found, no match, multiple matches, invalid regex
- [ ] No new external dependencies (use `regex` crate already available in workspace, or std pattern matching)
