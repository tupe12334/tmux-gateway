# Remove tokio dependency from core crate to achieve a truly pure functional core

The core crate (tmux-gateway-core) describes itself as "no networking" but depends on tokio for spawn_blocking. This violates the functional core / imperative shell principle: the domain core should contain only pure, synchronous logic with no runtime dependency.

## Problem
- Cargo.toml lists tokio = { version = "1", features = ["rt"] } as a core dependency
- Every domain function is async and calls tokio::task::spawn_blocking internally
- This forces consumers of the core crate to use tokio as their async runtime
- The core crate cannot be used in synchronous contexts, WASM, or other runtimes
- Existing issues #40, #47, #49, #57 address pieces of this but none target removing tokio from the core crate entirely

## What to do
1. Make all core domain functions synchronous (blocking) — they already run blocking tmux commands inside spawn_blocking
2. Move the async wrapper (spawn_blocking) to the application/shell layer in the main crate
3. Core crate exports: fn list_sessions() -> Result<Vec<TmuxSession>, TmuxError> (sync, blocking)
4. App crate wraps: async fn list_sessions() { spawn_blocking(|| core::list_sessions()).await }
5. Remove tokio from core Cargo.toml dependencies

## Acceptance criteria
- tmux-gateway-core has zero async runtime dependencies
- All core functions are synchronous and blocking
- The main crate handles async wrapping in the imperative shell layer
- Core crate is usable from any context (sync, tokio, async-std, WASM with mock backend)
