# Add DomainConfig value object for operation policies

## Problem

Domain operations have no shared configuration. Policies like operation timeout, maximum name length, minimum pane size for splitting, and validation strictness are either hardcoded as magic numbers or absent entirely:

- `validation.rs`: `MAX_NAME_LEN = 128` is a constant but not configurable
- `executor.rs`: no timeout on `spawn_blocking` calls
- `split_window`: no minimum pane dimension check
- No retry policy configuration

Configuration lives entirely in `src/main.rs` (env vars for ports, CORS, rate limits) with zero domain-level representation. The domain layer cannot adapt its behavior based on deployment context.

## Proposed change

Add a `DomainConfig` value object in `tmux-gateway-core`:

```rust
#[derive(Clone, Debug, PartialEq)]
pub struct DomainConfig {
    pub max_name_length: usize,
    pub operation_timeout: Duration,
    pub max_retry_attempts: u32,
    pub min_pane_width: u32,
    pub min_pane_height: u32,
    pub validation_strictness: ValidationStrictness,
}

#[derive(Clone, Copy, Debug, PartialEq, Eq)]
pub enum ValidationStrictness {
    Strict,   // reject anything suspicious
    Relaxed,  // accept valid tmux input, even unusual
}

impl DomainConfig {
    pub fn default() -> Self;
    pub fn validate(&self) -> Result<(), Vec<String>>;
}
```

Pure validation: `DomainConfig::validate()` checks internal consistency (e.g., timeout > 0, min dimensions > 0, max_name_length > 0).

Domain operations accept `&DomainConfig` (or it's injected via the executor). The imperative shell constructs `DomainConfig` from env vars in `main.rs`.

## Why this matters

- Makes hardcoded constants configurable without changing domain code
- Foundation for #87 (operation timeout), #124 (retry policy), #114 (resource limits)
- Different deployments can tune domain behavior (strict for production, relaxed for dev)
- Pure value object with self-validation — no I/O in domain layer
- API-agnostic: all three API layers share the same domain config

## Acceptance criteria

- [ ] `DomainConfig` struct with all policy fields and sensible defaults
- [ ] `ValidationStrictness` enum
- [ ] Pure `validate()` method that checks internal consistency
- [ ] `max_name_length` used by validation functions instead of hardcoded 128
- [ ] Unit tests for default config, custom config, invalid config detection
- [ ] No I/O — config is constructed by the imperative shell
