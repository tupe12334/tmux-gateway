# Add domain-level tracing spans to core operations

## Problem
Domain operations in tmux-gateway-core have no tracing instrumentation. All tracing is in the imperative shell (main.rs HTTP trace layer). When diagnosing slow operations or failures, there's no visibility into which domain operation was executing, what arguments it received, or how long the tmux subprocess took.

## Current state
Domain functions like list_sessions, kill_session, new_session etc. have no `#[tracing::instrument]` or manual span creation. The only tracing comes from the HTTP middleware, which logs method/path/status but not the domain operation details.

## Proposed change
Add `#[tracing::instrument]` to domain operations:

```rust
#[tracing::instrument(skip(self), fields(target))]
pub async fn kill_session(target: &str) -> Result<(), TmuxError> { ... }
```

This adds structured logging at the domain level — API-agnostic and visible regardless of which transport layer invoked the operation. The `tracing` crate is already a dependency.

Key spans to add:
- Operation name and arguments
- tmux subprocess duration (inside spawn_blocking)
- Error classification results

## Acceptance criteria
- [ ] All public domain operations have tracing::instrument
- [ ] Spans include operation name and relevant arguments
- [ ] tmux subprocess execution time is measurable from traces
- [ ] No new dependencies needed (tracing already in scope)
- [ ] Works with all API layers (traces appear regardless of transport)
