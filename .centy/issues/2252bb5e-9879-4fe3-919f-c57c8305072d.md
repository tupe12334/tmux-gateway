# Add conditional domain operations with precondition predicates

## Problem

Many gateway automation workflows need conditional logic: "kill the window only if it has no active process", "create a session only if one with that name doesn't exist", "split the pane only if the window has fewer than 4 panes." Currently, clients must make multiple round-trips (query state, decide, execute) with race conditions between the check and the action.

Tmux has `if-shell` for conditional execution, but it's shell-oriented and doesn't map cleanly to the gateway's typed domain model. A domain-level conditional operation abstraction would be more powerful and type-safe.

## Proposed change

Add a predicate-based conditional execution layer in the domain:

```rust
/// A predicate that can be evaluated against the current tmux state.
pub enum DomainPredicate {
    SessionExists(String),
    SessionNotExists(String),
    WindowCount { session: String, op: CompareOp, count: u32 },
    PaneCount { target: String, op: CompareOp, count: u32 },
    PaneContentMatches { target: String, pattern: String },
    PaneZoomed(String),
    Custom(Box<dyn Fn(&dyn TmuxExecutor) -> BoxFuture<'_, Result<bool, TmuxError>> + Send + Sync>),
}

pub enum CompareOp { Lt, Le, Eq, Ge, Gt }

/// Execute an operation only if the predicate holds. Returns Ok(None) if the predicate was false.
pub async fn execute_if<F, T>(
    executor: &impl TmuxExecutor,
    predicate: &DomainPredicate,
    operation: F,
) -> Result<Option<T>, TmuxError>
where
    F: FnOnce() -> BoxFuture<'_, Result<T, TmuxError>>,
```

This atomically evaluates the predicate and executes the operation, reducing race windows compared to client-side check-then-act patterns.

## Why this matters

- Reduces check-then-act race conditions by keeping predicate evaluation and execution close together
- Eliminates unnecessary API round-trips for conditional workflows
- Enables richer orchestration patterns without client-side state management
- Complements #140 (precondition validation) but focuses on runtime state predicates rather than input validation
- API-agnostic: any API can expose conditional execution semantics

## Acceptance criteria

- [ ] `DomainPredicate` enum with common tmux state predicates
- [ ] `CompareOp` for numeric comparisons
- [ ] `execute_if` function that evaluates predicate before running operation
- [ ] `SessionExists` and `SessionNotExists` predicates implemented
- [ ] `WindowCount` and `PaneCount` predicates implemented
- [ ] Unit tests with mock executor for each predicate variant
- [ ] Test that operation is NOT called when predicate is false
- [ ] Documentation on race condition reduction (not elimination)
