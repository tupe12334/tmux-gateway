# Separate tmux I/O adapter from domain operations

## Problem

Every domain operation (kill_session, new_session, list_sessions, etc.) directly calls tmux_interface inside spawn_blocking. The domain core has a hard dependency on the tmux_interface crate and real process execution. This makes the entire domain layer untestable without a running tmux server and violates the imperative shell / functional core separation.

## Current state

Each operation module (kill_session.rs, new_session.rs, etc.) imports tmux_interface types directly and calls Tmux::with_command(…).output() inside spawn_blocking. There is no abstraction boundary between “what to do” (domain) and “how to talk to tmux” (infrastructure).

## Proposed change

Introduce a TmuxAdapter trait (or TmuxExecutor) in the core crate that abstracts the tmux I/O boundary:

````rust
pub trait TmuxExecutor: Send + Sync {
    fn execute(&self, command: &str, args: &[&str]) -> Result<String, TmuxError>;
    fn execute_checked(&self, command: &str, args: &[&str]) -> Result<(), TmuxError>;
}
````

* A real implementation wraps tmux_interface (imperative shell)
* A test implementation returns canned responses (enables pure unit tests)
* Domain operations take `&impl TmuxExecutor` instead of calling tmux_interface directly

This enables:

1. Unit testing domain logic without tmux installed
1. Mocking tmux responses for edge case testing
1. Potential future support for remote tmux over SSH (different executor)

## Acceptance criteria

* [ ] TmuxExecutor trait defined in core crate
* [ ] Real implementation wrapping tmux_interface
* [ ] At least one domain operation refactored to use the executor
* [ ] Unit test demonstrating testing with a mock executor
* [ ] tmux_interface dependency is isolated to the executor implementation, not spread across all modules
