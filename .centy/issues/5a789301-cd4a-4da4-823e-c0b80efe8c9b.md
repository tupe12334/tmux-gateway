# Preserve validation error context when converting to TmuxError

## Problem

When a ValidationError is converted to TmuxError via the From impl, it becomes TmuxError::InvalidTarget(e.to_string()). This flattens the structured validation error into a string, losing the distinction between EmptyInput, InvalidSessionName, and InvalidTarget. API layers cannot provide specific error guidance (e.g., "session name is too long" vs "target contains invalid characters").

## Current state

In error.rs:

````rust
impl From<crate::validation::ValidationError> for TmuxError {
    fn from(e: crate::validation::ValidationError) -> Self {
        Self::InvalidTarget(e.to_string())
    }
}
````

All validation failures become `InvalidTarget("...")` regardless of the original type.

## Proposed change

Option A: Add a `Validation(ValidationError)` variant to TmuxError that wraps the original error:

````rust
#[error("validation error: {0}")]
Validation(#[from] ValidationError),
````

Option B: Add more specific TmuxError variants that match validation contexts:

````rust
InvalidSessionName(String),
InvalidWindowName(String),
EmptyInput { field: String },
````

Option A is simpler and preserves the full validation context. API layers can match on the inner ValidationError for specific responses.

## Acceptance criteria

* [ ] ValidationError context is preserved in TmuxError
* [ ] API layers can distinguish empty input from invalid characters from too-long names
* [ ] Existing error messages remain human-readable
* [ ] from_stderr classification still works (separate code path)
