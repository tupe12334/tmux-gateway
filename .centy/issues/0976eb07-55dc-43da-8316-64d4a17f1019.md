# Add operation precondition validation framework for domain operations

## Problem

Domain operations validate input format (name length, target syntax) but don't validate semantic preconditions. For example:

- `rename_session("foo", "foo")` — renaming to the same name is a no-op, but tmux is still called
- `kill_window` on the last window of a session kills the session — callers aren't warned
- `split_window` on a 1x1 pane will fail — but we don't check dimensions first
- `new_window` in a non-existent session fails with a confusing error

Issue #85 covers preconditions for destructive operations and issue #109 covers structural invariants, but there's no general framework for expressing and checking preconditions before any operation executes.

## Proposed change

Add a `preconditions` module in `tmux-gateway-core` with a general validation framework:

```rust
#[derive(Debug, Clone)]
pub struct PreconditionViolation {
    pub operation: String,
    pub condition: String,
    pub suggestion: String,
}

/// Pure: check preconditions for rename operations
pub fn check_rename_preconditions(
    current_name: &str,
    new_name: &str,
) -> Result<(), PreconditionViolation>;

/// Pure: check if a window can be killed safely
pub fn check_kill_window_preconditions(
    session: &TmuxSession,
    window_count: u32,
) -> Result<(), PreconditionViolation>;

/// Pure: check if a pane can be split given its dimensions
pub fn check_split_preconditions(
    pane: &TmuxPane,
    direction: SplitDirection,
    min_pane_size: u32,
) -> Result<(), PreconditionViolation>;

/// Pure: check if a session exists before operating on it
pub fn check_session_exists_precondition(
    target: &str,
    sessions: &[TmuxSession],
) -> Result<(), PreconditionViolation>;
```

Add a `TmuxError::PreconditionFailed` variant that maps to HTTP 422 / gRPC FAILED_PRECONDITION, distinct from validation errors (400) and not-found errors (404).

The imperative shell decides whether to run precondition checks (they require fetching current state), but the checks themselves are pure functions.

## Why this matters

- Better error messages: "cannot rename session 'foo' to 'foo' — name is unchanged" vs tmux's cryptic output
- Prevents unnecessary tmux calls for operations that would be no-ops or guaranteed failures
- Foundation for issues #85 (destructive preconditions) and #109 (structural invariants)
- All checks are pure functions — fully testable without tmux
- HTTP 422 / gRPC FAILED_PRECONDITION gives clients actionable error semantics

## Acceptance criteria

- [ ] `PreconditionViolation` struct with operation, condition, and suggestion fields
- [ ] `TmuxError::PreconditionFailed` variant with HTTP 422 / gRPC FAILED_PRECONDITION mapping
- [ ] Pure precondition check functions for rename, kill-window, split, and session-exists
- [ ] Each check returns a helpful suggestion message
- [ ] Unit tests for all precondition scenarios (pass and fail cases)
- [ ] At least 2 domain operations wired to run precondition checks before execution
