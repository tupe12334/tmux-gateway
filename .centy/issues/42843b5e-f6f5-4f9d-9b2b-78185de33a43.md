# Introduce validated newtypes for domain identifiers (SessionName, WindowTarget, PaneTarget)

The domain layer uses raw strings for all identifiers — session names, window targets ("session:window"), and pane targets ("session:window.pane"). This violates the Parse Don't Validate principle and makes it easy to pass the wrong string type to the wrong operation.

## Problem

- `new_session(name: &str)` accepts any string — callers can accidentally pass a window target
- `kill_window(target: &str)` and `rename_session(target: &str)` use the same `&str` type despite requiring completely different formats
- Validation happens inside each operation, not at construction time — invalid values travel through the system before being caught
- Target format knowledge ("session:window.pane") is spread across validation.rs, operation files, and API layers

## Solution

Create newtypes that can only be constructed through validation:

```rust
pub struct SessionName(String);      // validated: 1–128 chars, [a-zA-Z0-9._-]
pub struct WindowTarget(String);     // validated: "session:window" format
pub struct PaneTarget(String);       // validated: "session:window.pane" format
```

Each type should:
1. Implement `TryFrom<&str>` or a `parse()` method that returns `Result<Self, ValidationError>`
2. Implement `AsRef<str>` and `Display` for downstream use
3. Provide accessors (e.g., `PaneTarget::session()`, `PaneTarget::window()`, `PaneTarget::pane()`)
4. Be used in all domain operation signatures instead of raw `&str`

## Impact

- Compile-time prevention of target format confusion
- Validation happens once at construction, not in every operation
- API layers can parse input into domain types at the boundary
- Enables removing duplicated validation calls from operation functions
