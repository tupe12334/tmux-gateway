# Add domain model for tmux hook subscriptions and event-driven notifications

tmux has a built-in hook system (set-hook, show-hooks) that fires events on lifecycle changes: after-new-session, after-kill-window, window-linked, pane-exited, etc. The domain has no representation for these hooks, missing an opportunity for event-driven architecture.

## Problem
- Issue #57 proposes domain event types for operation notifications, but those are emitted by the gateway itself
- tmux-native hooks are a separate concept — they fire asynchronously from the tmux server
- No way to register callbacks or queries for tmux lifecycle events through the gateway
- Event-driven architectures (e.g. auto-cleanup on session close, audit logging) require hook support

## Proposed solution
Add domain operations for tmux hook management:

\`\`\`rust
pub struct TmuxHook {
    pub name: String,     // e.g. "after-new-session", "window-linked"
    pub command: String,  // tmux command to run when hook fires
}

/// List all hooks for a given session (or global hooks).
pub async fn list_hooks(session: Option<&str>) -> Result<Vec<TmuxHook>, TmuxError>

/// Set a hook at session or global level.
pub async fn set_hook(
    session: Option<&str>,
    hook_name: &str,
    command: &str,
) -> Result<(), TmuxError>

/// Remove a hook.
pub async fn unset_hook(
    session: Option<&str>,
    hook_name: &str,
) -> Result<(), TmuxError>
\`\`\`

Validate hook names against known tmux hook events. Validate commands for safety.

## Scope
- New \`hooks.rs\` module in \`tmux-gateway-core\`
- \`TmuxHook\` domain model
- Hook name validation (known event names only)
- Add to \`TmuxCommands\` trait
- Complements #57 (domain events emitted by gateway) with tmux-native events
