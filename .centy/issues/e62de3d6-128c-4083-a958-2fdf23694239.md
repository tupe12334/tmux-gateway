# Add fluent domain operation composition with automatic rollback

`create_session_with_windows` demonstrates the composite operation pattern with manual rollback. As more composite operations are added (#93 batch ops, #139 lifecycle, #110 cross-session), each will need its own rollback logic. A reusable composition framework eliminates duplicated error recovery code.

## Problem

- `create_session_with_windows` has hand-written rollback (kill session on window creation failure)
- Each new composite operation must re-implement rollback/cleanup logic
- No way to declare "if step 3 fails, undo steps 1 and 2"
- Batch operations (#93) will face the same problem at scale
- Partial failure handling is ad-hoc and inconsistent across operations

## Solution

Add a domain operation composer with declarative rollback:

```rust
pub struct OperationChain {
    steps: Vec<ChainStep>,
}

struct ChainStep {
    name: &'static str,
    execute: Box<dyn FnOnce() -> BoxFuture<Result<(), TmuxError>>>,
    rollback: Option<Box<dyn FnOnce() -> BoxFuture<Result<(), TmuxError>>>>,
}

impl OperationChain {
    pub fn new() -> Self { Self { steps: vec![] } }

    pub fn step(mut self, name: &'static str, execute: impl ..., rollback: impl ...) -> Self {
        self.steps.push(ChainStep { name, execute, rollback: Some(rollback) });
        self
    }

    /// Execute all steps; on failure, rollback completed steps in reverse order
    pub async fn execute(self) -> Result<ChainResult, TmuxError> {
        let mut completed = vec![];
        for step in self.steps {
            match (step.execute)().await {
                Ok(()) => completed.push(step),
                Err(e) => {
                    // Rollback completed steps in reverse
                    for s in completed.into_iter().rev() {
                        if let Some(rb) = s.rollback { let _ = (rb)().await; }
                    }
                    return Err(e);
                }
            }
        }
        Ok(ChainResult { steps_completed: completed.len() })
    }
}
```

## Impact

- Eliminates duplicated rollback logic across composite operations
- Declarative error recovery — each step defines its own undo
- Foundation for batch operations (#93) and lifecycle management (#139)
- Consistent partial failure handling across all composite workflows
- Pure composition logic is testable without tmux
