# Add domain-level pagination for list operations

## Problem

The gateway's list operations (`list_sessions`, `list_windows`, `list_panes`) return all results unconditionally. In production environments with many sessions (CI/CD systems, multi-tenant servers), this creates unbounded response sizes and unnecessary tmux subprocess output parsing.

There is no way for clients to request "give me sessions 10-20" or "give me the first 5 windows." Every list call returns everything, which doesn't scale.

## Proposed change

Add a `Pagination` value object and paginated variants of list operations:

```rust
#[derive(Debug, Clone, Default)]
pub struct Pagination {
    /// Number of items to skip (0-based offset)
    pub offset: usize,
    /// Maximum number of items to return (None = unlimited)
    pub limit: Option<usize>,
}

#[derive(Debug, Clone)]
pub struct PaginatedResult<T> {
    pub items: Vec<T>,
    /// Total count of items before pagination was applied
    pub total: usize,
    /// Whether more items exist beyond the returned page
    pub has_more: bool,
}

pub async fn list_sessions_paginated(
    executor: &impl TmuxExecutor,
    pagination: &Pagination,
) -> Result<PaginatedResult<TmuxSession>, TmuxError>;

pub async fn list_windows_paginated(
    executor: &impl TmuxExecutor,
    session: &str,
    pagination: &Pagination,
) -> Result<PaginatedResult<TmuxWindow>, TmuxError>;

pub async fn list_panes_paginated(
    executor: &impl TmuxExecutor,
    target: &str,
    pagination: &Pagination,
) -> Result<PaginatedResult<TmuxPane>, TmuxError>;
```

Implementation: the domain layer fetches all results from tmux (tmux has no native pagination), then applies offset/limit in Rust. The `total` field tells clients how many items exist overall. This is efficient because tmux output parsing is fast and the bottleneck is typically network transfer, not parsing.

## Why this matters

- Prevents unbounded response sizes in production environments
- Enables efficient UI pagination (tables, infinite scroll) without over-fetching
- `PaginatedResult<T>` is a reusable generic value object — any future list operation gets pagination for free
- Existing non-paginated operations remain unchanged (backward compatible)
- API-agnostic: REST uses query params, GraphQL uses arguments, gRPC uses message fields — all backed by the same domain pagination

## Acceptance criteria

- [ ] `Pagination` value object with offset and optional limit
- [ ] `PaginatedResult<T>` with items, total count, and has_more flag
- [ ] `list_sessions_paginated` function
- [ ] `list_windows_paginated` function
- [ ] `list_panes_paginated` function
- [ ] Default pagination (offset=0, limit=None) returns all items
- [ ] Offset beyond total returns empty items with correct total
- [ ] Unit tests with mock executor for each paginated operation
- [ ] Existing non-paginated operations unmodified
