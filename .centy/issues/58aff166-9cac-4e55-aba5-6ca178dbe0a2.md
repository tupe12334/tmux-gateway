# Add hierarchical domain model with aggregate relationships

## Problem

The domain model is flat: TmuxSession has `windows: u32` (a count), not `windows: Vec<TmuxWindow>`. TmuxWindow has `panes: u32`, not `panes: Vec<TmuxPane>`. There is no way to get a complete session view with its windows and panes in a single domain operation.

In DDD terms, TmuxSession should be an aggregate root with navigable relationships to its child entities (windows and panes). Currently, getting a full session picture requires three separate calls: `list_sessions()`, `list_windows(session)`, `list_panes(window)`.

## Current state

```rust
pub struct TmuxSession {
    pub name: String,
    pub windows: u32,  // just a count
    pub created: String,
    pub attached: bool,
}

pub struct TmuxWindow {
    pub index: u32,
    pub name: String,
    pub panes: u32,  // just a count
    pub active: bool,
}
```

## Proposed changes

1. Add a `SessionDetail` aggregate type that includes nested windows and panes:
```rust
pub struct SessionDetail {
    pub session: TmuxSession,
    pub windows: Vec<WindowDetail>,
}

pub struct WindowDetail {
    pub window: TmuxWindow,
    pub panes: Vec<TmuxPane>,
}
```

2. Add a domain function `get_session_detail(name: &str) -> Result<SessionDetail, TmuxError>` that composes list_windows and list_panes into a complete aggregate

3. Keep the flat types for list operations (listing all sessions doesn't need full detail)

This follows the DDD pattern of having both summary and detail views of aggregates, and enables API layers to serve rich queries (e.g., GraphQL nested queries) from a single domain call.

## Acceptance criteria

- [ ] SessionDetail and WindowDetail aggregate types exist in core crate
- [ ] get_session_detail composes existing operations into a hierarchical view
- [ ] Flat types (TmuxSession, TmuxWindow) remain for list operations
- [ ] All API layers can use the aggregate for rich queries
- [ ] Unit tests for aggregate composition
