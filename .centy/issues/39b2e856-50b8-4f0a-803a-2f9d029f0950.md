# Add domain precondition checks before destructive and mutating operations

Domain operations like rename_session, kill_window, and send_keys pass targets directly to tmux without checking preconditions. When tmux returns an error, it gets classified from stderr text. The domain layer should express and enforce its own business rules.

## Problem

- rename_session does not verify the session exists before attempting rename — relies on tmux stderr for error
- kill_window does not check that the target window exists — gets a generic error from tmux
- send_keys to a non-existent pane only fails at the tmux level, not the domain level
- The domain layer has no way to express business rules like "cannot kill the last window in a session"
- Error messages from tmux are less informative than domain-level precondition failures

## Solution

Add precondition checks as pure domain functions that run before the tmux command:

```rust
pub async fn rename_session(target: &str, new_name: &str) -> Result<(), TmuxError> {
    validation::validate_session_target(target)?;
    validation::validate_session_name(new_name)?;
    ensure_session_exists(target).await?;     // precondition check
    // ... execute tmux command
}
```

Key preconditions to add:
1. **Session exists** — before rename_session, kill_session
2. **Window exists** — before rename_window, kill_window
3. **Session does not exist** — before new_session (duplicate prevention)
4. **Target pane exists** — before send_keys, capture_pane

These use existing domain queries (list_sessions, list_windows, list_panes) to verify state before mutation.

## Impact

- Domain layer expresses business rules explicitly
- Better error messages: "session 'foo' not found" before tmux is even invoked
- Decouples error semantics from tmux stderr parsing
- All API layers benefit from clearer, earlier error reporting
