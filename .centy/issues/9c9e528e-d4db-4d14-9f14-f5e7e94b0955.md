# Add tmux control mode executor for persistent, efficient communication

## Problem

Every domain operation currently spawns a new `tmux` subprocess via `RealTmuxExecutor::execute()` (using `std::process::Command`). For high-frequency operations (monitoring, orchestration, rapid command sequences), this creates significant overhead:

1. Process spawn overhead (~2-5ms per command on macOS/Linux)
2. No connection reuse between commands
3. Cannot receive asynchronous notifications from tmux
4. Sequential request-response only — no streaming

Tmux's **control mode** (`tmux -C` or `tmux attach -t <session> -C`) provides a persistent, line-oriented protocol over stdin/stdout that eliminates per-command process spawning and enables push notifications.

## Proposed change

Add a `ControlModeExecutor` as an alternative `TmuxExecutor` implementation:

```rust
/// A TmuxExecutor that communicates via tmux control mode over a persistent connection.
/// Commands are sent as text lines; responses are parsed from the %begin/%end/%error protocol.
pub struct ControlModeExecutor {
    // Internal: tokio ChildStdin/ChildStdout for the persistent tmux -C process
}

impl ControlModeExecutor {
    /// Start a control mode connection to the given session.
    pub async fn connect(session: &str) -> Result<Self, TmuxError>;

    /// Gracefully close the control mode connection.
    pub async fn disconnect(self) -> Result<(), TmuxError>;

    /// Check if the control mode connection is still alive.
    pub fn is_connected(&self) -> bool;
}

impl TmuxExecutor for ControlModeExecutor {
    async fn execute(&self, args: &[&str]) -> Result<TmuxOutput, TmuxError> {
        // Send command over stdin, parse %begin/%end/%error response from stdout
    }
}
```

The domain layer remains unchanged — it depends on `&impl TmuxExecutor`, so the imperative shell (API layer) chooses which executor to use. This follows the ports & adapters pattern already established.

## Why this matters

- **Performance**: 10-50x reduction in per-command latency for rapid operation sequences
- **Foundation for streaming**: Control mode emits `%output`, `%window-renamed`, `%session-changed` notifications — future issues (#57, #143) can build on this
- **No domain changes needed**: Swapping executor at the API layer is transparent to all domain operations
- **Production readiness**: High-throughput use cases (monitoring dashboards, CI/CD automation) need efficient tmux communication
- Complements #137 (command pipeline optimization) with a persistent-connection approach

## Acceptance criteria

- [ ] `ControlModeExecutor` struct implementing `TmuxExecutor` trait
- [ ] `connect(session)` starts `tmux -C attach -t <session>` as a persistent child process
- [ ] `disconnect()` sends `detach` command and waits for clean shutdown
- [ ] `is_connected()` health check
- [ ] `execute()` sends command text, parses `%begin`/`%end`/`%error` response framing
- [ ] Handles concurrent command submissions (internal serialization or multiplexing)
- [ ] Reconnection strategy on connection loss (return error, caller decides)
- [ ] Unit tests with mock stdin/stdout streams
- [ ] Integration test connecting to a real tmux session (behind feature flag or ignored by default)
- [ ] No changes to domain layer or existing `RealTmuxExecutor`
