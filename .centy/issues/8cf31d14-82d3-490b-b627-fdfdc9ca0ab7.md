# Add domain cache port for reducing redundant tmux command calls

## Problem

Every domain query (list_sessions, list_windows, list_panes) spawns a blocking tmux process, even when the data was fetched milliseconds ago. In common workflows like "list sessions, then get windows for each session, then get panes for each window", the session list is stale-proof but still fetched fresh each time.

This matters for:
- Composite operations (#46) that make multiple queries internally
- Health checks that poll every 5 seconds
- API consumers making rapid successive requests
- GraphQL resolvers that may trigger N+1 tmux calls

## Proposed change

Add a `CachePort` trait in `tmux-gateway-core` following ports & adapters:

```rust
use std::time::Duration;

pub trait CachePort: Send + Sync {
    /// Get cached value if fresh enough
    fn get<T: Clone + Send + 'static>(&self, key: &str) -> Option<T>;

    /// Store value with TTL
    fn put<T: Clone + Send + 'static>(&self, key: &str, value: T, ttl: Duration);

    /// Invalidate a specific key
    fn invalidate(&self, key: &str);

    /// Invalidate all entries matching a prefix (e.g., "session:" after a mutation)
    fn invalidate_prefix(&self, prefix: &str);
}

/// No-op cache for when caching is disabled
pub struct NoCache;
impl CachePort for NoCache {
    fn get<T: Clone + Send + 'static>(&self, _: &str) -> Option<T> { None }
    fn put<T: Clone + Send + 'static>(&self, _: &str, _: T, _: Duration) {}
    fn invalidate(&self, _: &str) {}
    fn invalidate_prefix(&self, _: &str) {}
}
```

Domain operations check the cache before calling tmux. Mutations invalidate relevant cache entries. The imperative shell provides the concrete implementation (in-memory LRU, or NoCache for simplicity).

Cache key conventions:
- `sessions:list` — list_sessions result
- `windows:{session}` — list_windows for a session
- `panes:{target}` — list_panes for a target

Mutations invalidate:
- `new_session` / `kill_session` / `rename_session` → invalidate `sessions:*`
- `new_window` / `kill_window` / `rename_window` → invalidate `windows:{session}` and `sessions:list`
- `split_window` / `kill_pane` → invalidate `panes:{target}` and `windows:{session}`

## Why this matters

- Reduces tmux process spawns in high-traffic scenarios
- Makes composite operations (#46) significantly faster
- GraphQL N+1 resolution becomes O(1) for cached data
- Follows hexagonal architecture: domain defines port, infrastructure provides adapter
- NoCache means zero behavior change when caching is disabled
- API-agnostic: all three API layers benefit equally

## Acceptance criteria

- [ ] `CachePort` trait in core crate with get/put/invalidate/invalidate_prefix
- [ ] `NoCache` default implementation
- [ ] At least `list_sessions` and `list_windows` check cache before calling tmux
- [ ] Mutation operations invalidate relevant cache entries
- [ ] Cache is optional — NoCache by default, configurable TTL
- [ ] Unit tests verify cache hit/miss behavior and invalidation correctness
- [ ] No concrete cache library dependency in core crate
