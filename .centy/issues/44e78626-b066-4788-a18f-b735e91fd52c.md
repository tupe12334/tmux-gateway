# Add composite domain operations for multi-step tmux workflows

## Problem

The domain layer only exposes atomic tmux operations (create one session, kill one window, etc.). Common workflows require multiple sequential calls that should be atomic at the domain level:

* Create a session with N pre-configured windows
* Kill a session and all its windows/panes (tmux does this implicitly, but there’s no domain-level acknowledgment)
* Move a window between sessions
* Swap panes within a window

These compositions currently must be orchestrated by each API layer independently, leading to duplicated logic and inconsistent behavior.

## Proposed change

Add composite domain functions in tmux-gateway-core:

````rust
pub async fn create_session_with_windows(name: &str, window_names: &[&str]) -> Result<TmuxSession, TmuxError>
pub async fn swap_panes(pane_a: &str, pane_b: &str) -> Result<(), TmuxError>
pub async fn move_window(source: &str, destination_session: &str) -> Result<(), TmuxError>
````

These encapsulate multi-step workflows as single domain operations, ensuring consistency across all API layers.

## Acceptance criteria

* [ ] At least create_session_with_windows implemented
* [ ] Validation applied to all inputs
* [ ] Error handling for partial failures (e.g., session created but window creation fails)
* [ ] Added to TmuxCommands trait
* [ ] Available from all API layers without duplication
