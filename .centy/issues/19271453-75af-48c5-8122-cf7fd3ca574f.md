# Abstract tmux infrastructure behind a TmuxBackend trait for testability

Core domain functions directly import and call tmux_interface::Tmux, coupling business logic to infrastructure. This makes unit testing domain logic impossible without a running tmux server.

## Problem
- Every core function (list_sessions, new_session, kill_session, etc.) directly uses tmux_interface::{Tmux, ListSessions, NewSession, ...}
- tokio::task::spawn_blocking lives inside domain functions
- Cannot test domain logic (validation + error classification) without tmux installed
- Cannot swap tmux backends (local, remote SSH, container-based)

## What to do
Define a TmuxBackend trait in the core crate that abstracts raw tmux command execution:

trait TmuxBackend: Send + Sync {
    fn run_command(&self, args: &[&str]) -> Result<CommandOutput, TmuxError>;
}

struct CommandOutput { pub stdout: String, pub stderr: String, pub success: bool }

Provide a default LocalTmuxBackend implementation using tmux_interface. Domain functions take &dyn TmuxBackend (or generic T: TmuxBackend) so tests can inject a MockTmuxBackend that returns canned stdout/stderr.

## Acceptance criteria
- Core domain functions accept a backend parameter instead of hardcoding tmux_interface
- A MockTmuxBackend exists for testing
- Unit tests can verify domain logic (parsing, error classification) without tmux installed
- The existing LocalTmuxBackend preserves current behavior
