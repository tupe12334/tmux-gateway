# Add send_keys input validation to prevent unsafe key sequences

The `send_keys` domain operation validates the target pane but performs no validation on the keys themselves. This is an input validation gap at the domain boundary.

## Problem
- `send_keys(target, keys)` validates target via `validate_pane_target()` but accepts any `&[String]` for keys
- Keys are passed directly to tmux without sanitization
- While tmux itself handles key interpretation, the domain should enforce basic safety:
  - Reject empty individual key strings
  - Reject excessively long key sequences (potential abuse)
  - Optionally validate known tmux key names (Enter, Escape, C-c, etc.)

## Proposed solution
Add key validation to the domain validation module:

```rust
pub fn validate_key_sequence(keys: &[String]) -> Result<(), ValidationError> {
    if keys.is_empty() {
        return Err(ValidationError::EmptyInput { field: "keys" });
    }
    for key in keys {
        if key.is_empty() {
            return Err(ValidationError::EmptyInput { field: "key" });
        }
        if key.len() > MAX_KEY_LEN {
            return Err(ValidationError::InvalidKey { reason: "..." });
        }
    }
    Ok(())
}
```

Add a new `ValidationError::InvalidKey { reason: String }` variant.

## Scope
- Add `validate_key_sequence()` to `validation.rs`
- Add `InvalidKey` variant to `ValidationError`
- Call validation in `send_keys` before executing
- Add unit tests for key validation edge cases
