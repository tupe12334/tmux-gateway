# Add validation security hardening for null bytes, control characters, and homoglyphs

## Problem

Validation in `validation.rs` checks for shell metacharacters and enforces a character whitelist (alphanumeric, hyphen, underscore, dot). However, several attack vectors are not addressed:

1. **Null bytes**: `"test\0injection"` — the null byte could truncate the string at the C boundary when passed to tmux, causing the actual name to differ from what was validated
2. **Control characters**: `"test\x01\x02"` — non-printable characters pass alphabetic checks in some locales
3. **Homoglyph attacks**: Cyrillic "а" (U+0430) looks identical to ASCII "a" (U+0061) but is a different character — could cause confusion in session naming
4. **Zero-width characters**: U+200B (zero-width space), U+FEFF (BOM) — invisible characters that make names appear identical but aren't

Current validation uses `c.is_alphanumeric()` which accepts ALL Unicode alphanumerics, including non-ASCII characters. This means Cyrillic, Chinese, Arabic characters all pass validation.

## Proposed change

Add security-focused validation functions in `validation.rs`:

```rust
/// Pure: check if string contains null bytes
pub fn contains_null_bytes(s: &str) -> bool;

/// Pure: check if string contains non-printable control characters (U+0000-U+001F, U+007F-U+009F)
pub fn contains_control_chars(s: &str) -> bool;

/// Pure: check if string contains only ASCII characters (safe for tmux)
pub fn is_ascii_safe(s: &str) -> bool;

/// Pure: check if string contains zero-width or invisible characters
pub fn contains_invisible_chars(s: &str) -> bool;

/// Pure: normalize a name by stripping invisible characters and validating ASCII-only
pub fn sanitize_name(s: &str) -> Result<String, ValidationError>;
```

Update `validate_session_name` and `validate_window_name` to:
1. Reject null bytes (always)
2. Reject control characters (always)
3. Optionally reject non-ASCII (configurable via `DomainConfig.validation_strictness`)
4. Warn on invisible characters

Add a new `ValidationError` variant:
```rust
SecurityViolation { field: String, details: String }
```

## Why this matters

- Null byte injection is a well-known attack vector at C/system boundaries
- Tmux is a C program — null bytes in names could cause undefined behavior
- Homoglyph confusion could lead to session impersonation (two sessions that look identical but aren't)
- All checks are pure functions — zero I/O, fully testable
- Issue #113 (authorization model) needs secure identity; names must be unambiguous

## Acceptance criteria

- [ ] `contains_null_bytes`, `contains_control_chars`, `is_ascii_safe`, `contains_invisible_chars` as pure functions
- [ ] `validate_session_name` and `validate_window_name` reject null bytes and control characters
- [ ] `ValidationError::SecurityViolation` variant for security-specific rejections
- [ ] Unit tests for: null bytes, control chars (0x01-0x1F), zero-width space, BOM, Cyrillic homoglyphs
- [ ] Existing tests continue to pass (ASCII alphanumeric, hyphen, underscore, dot still valid)
- [ ] No external dependency — pure character checks using stdlib
