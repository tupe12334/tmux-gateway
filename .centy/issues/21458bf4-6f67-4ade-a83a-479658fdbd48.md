# Ensure both HTTP and gRPC servers start successfully before accepting traffic

The current startup spawns HTTP and gRPC servers independently via tokio::try_join!. If one fails to bind, the other still runs, leaving the server in a partial state.

## Problem
- If gRPC port bind fails, HTTP still serves (and vice versa)
- Port table shows 'free' status before actual bind happens (TOCTOU race)
- No readiness signal - clients can't tell when both servers are ready
- Server appears healthy via /health even if gRPC is down

## Fix
- Add a startup barrier: both servers must bind successfully before either accepts traffic
- Log clear error if either server fails to start
- Consider a readiness probe endpoint that checks both protocols
- Exit with clear error code if either server fails on startup
