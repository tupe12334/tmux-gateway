# Add domain event types for operation notifications

## Problem
The domain layer has no concept of events. Operations execute and return results, but there's no way to notify interested parties that something happened (e.g., "session created", "pane content changed"). This is a prerequisite for features like real-time streaming (#8) and will be needed by any API layer that wants to push updates to clients.

## Current state
All domain operations are request-response only. There's no event type, no event emission, and no way to subscribe to changes. Issue #8 (WebSocket streaming) has no domain foundation to build on.

## Proposed change
Define domain event types in tmux-gateway-core:

```rust
pub enum TmuxEvent {
    SessionCreated { name: String },
    SessionKilled { name: String },
    WindowCreated { session: String, name: String },
    WindowKilled { target: String },
    PaneCreated { target: String },
    PaneKilled { target: String },
    KeysSent { target: String },
    SessionRenamed { old_name: String, new_name: String },
    WindowRenamed { target: String, new_name: String },
}
```

Optionally, provide an event channel that domain operations emit into:
```rust
pub type EventSender = tokio::sync::broadcast::Sender<TmuxEvent>;
```

This keeps the event model in the domain layer. API layers can subscribe and forward events via their respective transport (WebSocket, gRPC streaming, GraphQL subscriptions).

## Acceptance criteria
- [ ] TmuxEvent enum defined in core crate
- [ ] Events are a domain concept — no transport/serialization concerns
- [ ] Design supports multiple subscribers (broadcast channel or similar)
- [ ] At least one domain operation emits an event as proof of concept
- [ ] Foundation for #8 (WebSocket streaming)
