# Make tmux server socket path injectable in domain operations

## Problem

All domain operations in `crates/tmux-gateway-core/src/` are implicitly coupled to the default tmux server socket. Every operation calls `Tmux::with_command(...)` which connects to the system-default tmux socket. There is no way to target a specific tmux server instance.

This violates the functional core / imperative shell principle: the tmux socket path is an infrastructure detail that is hardcoded into domain operations rather than being injected.

## Impact

- Cannot manage multiple tmux servers from a single gateway instance
- Cannot use separate tmux sockets for test isolation (tests share state via the global server)
- Cannot point the gateway at a remote tmux socket (e.g., via SSH forwarding)
- Domain operations are not pure — they implicitly depend on system state

## Current state

Every domain function directly uses the global socket:
```rust
Tmux::with_command(ListSessions::new().format(...))
    .output()  // always uses default socket
```

tmux supports `-S /path/to/socket` to target a specific server instance.

## Proposed changes

1. Add a `TmuxConnection` or `TmuxSocket` domain type that wraps the socket path:
```rust
pub struct TmuxConnection {
    socket_path: Option<PathBuf>,  // None = default socket
}
```

2. Thread connection through domain functions or make it part of a domain service struct
3. Update TmuxCommands trait to carry the connection context (e.g., `&self` already has it if impl holds the connection)
4. Default behavior (None) preserves current global socket behavior
5. Test helpers can create isolated tmux servers per test

## Acceptance criteria

- [ ] Domain operations can target a specific tmux socket path
- [ ] Default behavior (no socket specified) is unchanged
- [ ] TmuxCommands implementations receive connection context
- [ ] No breaking changes for existing API consumers when socket is not specified
- [ ] At least one test validates targeting a non-default socket
