# Add domain operations for window-level and pane-level options

Issue #59 covers session-level options, but tmux also supports window-level and pane-level options (e.g. \`remain-on-exit\`, \`synchronize-panes\`, \`pane-border-style\`). These are essential for fine-grained control of tmux behavior through the gateway.

## Problem
- No way to get or set window-specific options (e.g. \`automatic-rename\`, \`monitor-activity\`, \`synchronize-panes\`)
- No way to get or set pane-specific options (e.g. \`remain-on-exit\`, \`allow-rename\`)
- tmux has three option scopes: session, window, pane — only session is planned (#59)

## Proposed solution
Add domain operations for window and pane option management:

\`\`\`rust
// Window options
pub async fn get_window_option(target: &str, option: &str) -> Result<String, TmuxError>
pub async fn set_window_option(target: &str, option: &str, value: &str) -> Result<(), TmuxError>

// Pane options
pub async fn get_pane_option(target: &str, option: &str) -> Result<String, TmuxError>
pub async fn set_pane_option(target: &str, option: &str, value: &str) -> Result<(), TmuxError>
\`\`\`

Option names and values should be validated at the domain boundary:
- Option name: non-empty, alphanumeric + hyphens only (tmux option naming convention)
- Value: non-empty, safe characters (no shell metacharacters)
- Target: validated via existing \`validate_window_target\` / \`validate_pane_target\`

Consider a shared \`TmuxOption\` value object or \`OptionScope\` enum to unify with session options (#59):
\`\`\`rust
pub enum OptionScope { Session, Window, Pane }
\`\`\`

## Scope
- New modules: \`window_options.rs\`, \`pane_options.rs\` (or unified \`options.rs\`)
- Add option name/value validation to \`validation.rs\`
- Add to \`TmuxCommands\` trait
- Coordinate with #59 for consistent API design across all option scopes
