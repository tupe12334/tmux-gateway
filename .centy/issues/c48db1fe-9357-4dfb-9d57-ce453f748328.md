# Add query specification pattern for domain list operations

## Problem

List operations currently accept minimal parameters (session name, target). There is no composable way to express "list sessions created after X", "list windows with more than 3 panes", or "list panes running bash". Clients must fetch all results and filter client-side, which is wasteful and pushes domain logic into the API layer.

Issue #86 adds `find-window` for text search, but there's no general-purpose query mechanism for filtering and sorting domain entities by arbitrary field predicates.

## Proposed change

Add a specification/criteria pattern for domain queries:

```rust
/// A composable specification for filtering domain entities.
pub enum SessionSpec {
    All,
    NameEquals(String),
    NameContains(String),
    Attached,
    Detached,
    CreatedAfter(i64),       // Unix timestamp
    WindowCountAtLeast(u32),
    And(Box<SessionSpec>, Box<SessionSpec>),
    Or(Box<SessionSpec>, Box<SessionSpec>),
    Not(Box<SessionSpec>),
}

pub enum SortField {
    Name,
    Created,
    WindowCount,
}

pub enum SortOrder {
    Ascending,
    Descending,
}

pub struct ListQuery<S> {
    pub spec: S,
    pub sort_by: Option<SortField>,
    pub sort_order: SortOrder,
}

/// List sessions matching a specification.
pub async fn query_sessions(
    executor: &impl TmuxExecutor,
    query: &ListQuery<SessionSpec>,
) -> Result<Vec<TmuxSession>, TmuxError>;
```

Similar `WindowSpec` and `PaneSpec` enums for windows and panes. The implementation fetches all entities from tmux and applies the specification as a predicate filter in the domain layer.

## Why this matters

- Enables expressive queries without leaking filtering logic into API layers
- Specifications compose naturally: `And(Attached, WindowCountAtLeast(3))`
- Each API can map its native query syntax to the same domain specs
- Complements #166 (pagination) — filter first, then paginate
- Specifications are pure functions — fully testable without tmux
- Follows the Specification pattern from DDD tactical design

## Acceptance criteria

- [ ] `SessionSpec` enum with field predicates and logical combinators (And, Or, Not)
- [ ] `WindowSpec` and `PaneSpec` with appropriate field predicates
- [ ] `ListQuery<S>` combining spec with sort options
- [ ] `query_sessions` function applying spec as filter
- [ ] `query_windows` and `query_panes` functions
- [ ] Unit tests for each predicate variant
- [ ] Unit tests for And/Or/Not composition
- [ ] Sorting by each field in both orders
- [ ] Empty result when no entities match
