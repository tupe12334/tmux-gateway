# Add domain-level operation journal for audit trail

## Problem

There is no domain concept for recording what operations were performed against the tmux server. In production, operators need to answer questions like "who created this session?", "what commands were sent to this pane in the last hour?", and "what happened before this session died?".

Issue #57 covers domain events for real-time notifications, but there is no persistent, queryable journal of past operations.

## Proposed change

Add pure domain types for operation journaling in `tmux-gateway-core`:

```rust
pub struct OperationRecord {
    id: String,
    operation: OperationType,
    target: String,
    parameters: Vec<(String, String)>,
    outcome: OperationOutcome,
    timestamp: i64,
    duration_ms: u64,
}

pub enum OperationType {
    CreateSession, KillSession, RenameSession,
    CreateWindow, KillWindow, RenameWindow, SplitWindow,
    KillPane, SendKeys, CapturePane,
    // extensible for future operations
}

pub enum OperationOutcome {
    Success,
    Failed { error: String },
}
```

Pure functions on `OperationRecord` and collections:

- `OperationJournal` — an append-only in-memory log with max capacity
- `filter_by_target(target: &str) -> Vec<&OperationRecord>` — find ops on a target
- `filter_by_type(op: OperationType) -> Vec<&OperationRecord>` — find ops by kind
- `filter_by_time_range(from: i64, to: i64) -> Vec<&OperationRecord>` — time range query
- `last_n(n: usize) -> Vec<&OperationRecord>` — recent operations

The journal types are pure domain — the imperative shell (API layer) is responsible for actually recording entries and choosing persistence strategy (in-memory, file, database).

## Why this matters

- Production debugging requires knowing what happened and when
- Security auditing requires operation attribution
- Issue #119 (operation result envelope) provides per-operation metadata; the journal aggregates this over time

## Acceptance criteria

- [ ] `OperationRecord`, `OperationType`, `OperationOutcome` types in core crate
- [ ] `OperationJournal` with bounded capacity and query methods
- [ ] All query/filter methods are pure functions (no I/O)
- [ ] Unit tests for journal capacity eviction, filtering, time range queries
- [ ] No I/O or persistence logic in domain layer
