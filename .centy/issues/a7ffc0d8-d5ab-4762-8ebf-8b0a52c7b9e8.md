# Add domain-level command serialization for same-target operations

## Problem

When multiple API clients (or concurrent requests from a single client) issue operations against the same tmux session/window/pane simultaneously, race conditions can occur. For example:

1. Client A calls `rename_window("sess:0", "new-name")` while Client B calls `kill_window("sess:0")`
2. Client A calls `send_keys("sess:0.1", ["ls"])` while Client B calls `split_window("sess:0")` — the pane indices may shift
3. Two clients both call `new_session("test")` simultaneously — one gets `SessionAlreadyExists` non-deterministically

Tmux itself serializes commands internally, but the gateway's domain layer has no awareness of this — concurrent domain operations can interleave in unexpected ways.

## Proposed change

Add a domain-level command serialization mechanism using target-scoped locking:

```rust
/// Serializes operations targeting the same tmux entity.
/// Operations on different targets proceed concurrently.
pub struct OperationSerializer {
    // Internal: HashMap<String, tokio::sync::Mutex<()>> or similar per-target lock
}

impl OperationSerializer {
    pub fn new() -> Self;

    /// Execute an operation while holding the lock for the given target scope.
    /// Target scope is extracted from the target string (e.g., "sess:0.1" -> session "sess").
    pub async fn serialize<F, T>(
        &self,
        target_scope: &str,
        operation: F,
    ) -> Result<T, TmuxError>
    where
        F: FnOnce() -> BoxFuture<'_, Result<T, TmuxError>>;
}

/// Extract the serialization scope from a target string.
/// "sess:0.1" -> "sess" (session-level), or "sess:0" -> "sess:0" (window-level)
pub fn target_scope(target: &str, granularity: ScopeGranularity) -> String;

pub enum ScopeGranularity { Session, Window, Pane }
```

The imperative shell (API layer) wraps domain calls with the serializer. The domain itself remains pure — no locking awareness needed.

## Why this matters

- Prevents race conditions that cause confusing errors for API clients
- Session-level scope prevents the worst interleaving (pane index shifts during operations)
- Per-target scoping means unrelated sessions remain fully concurrent — no global bottleneck
- Pure infrastructure concern: domain functions stay stateless and lock-free
- Complements #151 (DomainConfig) with operational policy for concurrent access
- Necessary for production readiness when multiple clients or automated systems share a tmux server

## Acceptance criteria

- [ ] `OperationSerializer` struct with per-target locking
- [ ] `serialize()` acquires target-scoped lock before running operation
- [ ] `target_scope()` extracts scope at configurable granularity
- [ ] Different targets proceed concurrently (no global lock)
- [ ] Same-target operations are serialized (FIFO ordering)
- [ ] Lock cleanup: stale entries removed when no waiters remain
- [ ] Unit tests: concurrent operations on same target are serialized
- [ ] Unit tests: concurrent operations on different targets run in parallel
- [ ] No changes to domain layer functions
