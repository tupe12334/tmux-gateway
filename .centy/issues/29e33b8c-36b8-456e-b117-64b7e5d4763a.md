# Add domain-level concurrency guard for session-scoped mutations

Concurrent mutations targeting the same tmux session can cause race conditions. For example, two simultaneous rename_session calls on the same session, or a kill_session racing with a new_window on that session. The domain layer has no mechanism to coordinate concurrent access.

## Problem

- All API layers (REST, gRPC, GraphQL) can receive concurrent requests targeting the same session
- rename_session + rename_session on the same target can produce unpredictable results
- kill_session + new_window on the same session can race
- new_session with the same name from two concurrent requests may both succeed or one fails with unclear error
- The domain layer treats each operation as independent with no coordination

## Solution

Add a domain-level concurrency guard that serializes mutations per session:

```rust
use std::collections::HashMap;
use tokio::sync::Mutex;

pub struct SessionLock {
    locks: Mutex<HashMap<String, Arc<Mutex<()>>>>,
}

impl SessionLock {
    pub async fn acquire(&self, session: &str) -> SessionGuard {
        let lock = {
            let mut map = self.locks.lock().await;
            map.entry(session.to_string())
                .or_insert_with(|| Arc::new(Mutex::new(())))
                .clone()
        };
        let guard = lock.lock().await;
        SessionGuard { _guard: guard }
    }
}
```

Operations that mutate a session acquire the lock before executing:

```rust
pub async fn rename_session(lock: &SessionLock, target: &str, new_name: &str) -> Result<(), TmuxError> {
    let _guard = lock.acquire(target).await;
    // ... validation and execution
}
```

Read operations (list_sessions, list_windows, capture_pane) do NOT acquire the lock.

## Impact

- Prevents race conditions on concurrent session mutations
- Domain expresses concurrency as a first-class concern
- All API layers benefit automatically through the shared domain service
- Read operations remain fully concurrent (no contention)
