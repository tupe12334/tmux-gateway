# Add session locking domain operations for terminal security

tmux supports lock-session and lock-server commands that password-protect terminals, preventing unauthorized interaction with running sessions. The gateway has no domain operations for this security feature.

## Problem

- No way to lock/unlock sessions through the gateway
- Unattended terminals managed by the gateway remain accessible indefinitely
- Security policies may require automatic locking after inactivity
- tmux lock-session and lock-server are important security features with no domain representation
- Complements authorization (#113) with physical terminal protection

## Solution

Add domain operations for session and server locking:

```rust
/// Lock a specific session, requiring password to unlock
pub async fn lock_session(target: &str) -> Result<(), TmuxError> {
    validate_session_target(target)?;
    // Execute tmux lock-session -t target
}

/// Lock the entire tmux server
pub async fn lock_server() -> Result<(), TmuxError> {
    // Execute tmux lock-server
}

/// Domain model for lock configuration
pub struct LockConfig {
    pub lock_command: Option<String>,        // Custom lock program
    pub lock_after_seconds: Option<u32>,     // Auto-lock timeout
}

/// Set lock configuration for a session
pub async fn set_lock_config(
    target: &str,
    config: &LockConfig,
) -> Result<(), TmuxError> {
    validate_session_target(target)?;
    // Set lock-command and lock-after-time options
}
```

## Impact

- Enables security policies for gateway-managed terminals
- Complements domain authorization model (#113)
- Supports compliance requirements for terminal access control
- Allows programmatic lock/unlock for automated workflows
