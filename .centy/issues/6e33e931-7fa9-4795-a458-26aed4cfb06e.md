# Add if-shell conditional domain operation for predicate-based execution

tmux `if-shell` evaluates a shell command and conditionally executes a tmux command based on the result. This enables smart automation workflows where the gateway can make decisions based on system state. No existing issue covers this capability.

## Problem

- No way to conditionally execute tmux operations based on runtime state
- Clients must implement two-step check-then-act patterns manually (list, check, then act)
- TOCTOU races exist between checking state and acting on it (#67 covers idempotency, not conditional execution)
- Complex workflows require multiple round-trips between client and gateway
- tmux `if-shell` is a fundamental automation primitive with no domain representation

## Solution

Add a domain operation for conditional execution with a `ConditionalCommand` value object:

```rust
pub struct ConditionalCommand {
    /// Shell command whose exit code determines which branch to execute
    pub predicate: String,
    /// tmux command to execute if predicate succeeds (exit 0)
    pub if_true: String,
    /// Optional tmux command to execute if predicate fails (exit non-0)
    pub if_false: Option<String>,
}

pub async fn if_shell(
    target: &str,
    conditional: &ConditionalCommand,
) -> Result<(), TmuxError> {
    validate_session_target(target)?;
    validate_shell_predicate(&conditional.predicate)?; // Injection prevention
    // Execute: tmux if-shell -t target "predicate" "if_true" ["if_false"]
}
```

Security: validate predicates for injection prevention — disallow shell metacharacters, allow only safe commands like `test`, `grep -q`, `pgrep`.

## Impact

- Reduces client round-trips for conditional workflows
- Atomic check-and-act eliminates TOCTOU races
- Enables server-side automation logic
- Complements batch operations (#93) with conditional branching
