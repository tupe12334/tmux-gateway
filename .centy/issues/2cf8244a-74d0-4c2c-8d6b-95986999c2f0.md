# Consolidate spawn_blocking boilerplate with a domain executor helper

## Problem
Every domain operation in tmux-gateway-core follows the exact same pattern:
1. Clone string arguments
2. spawn_blocking(move || { ... })
3. Call tmux_interface
4. Check output.status.success()
5. Parse stderr with from_stderr on failure
6. Map JoinError to CommandFailed

This identical 15-line boilerplate is repeated in all 13 operation modules (kill_session.rs, new_session.rs, list_sessions.rs, etc.). It obscures the actual domain logic and makes each module harder to read.

## Current state
Every module has:
```rust
tokio::task::spawn_blocking(move || {
    let output = Tmux::with_command(...)
        .output()
        .map_err(|e| TmuxError::CommandFailed { command: "...", stderr: e.to_string() })?
        .into_inner();
    if !output.status.success() {
        let stderr = String::from_utf8_lossy(&output.stderr);
        return Err(TmuxError::from_stderr("...", &stderr, &target));
    }
    // actual logic
})
.await
.map_err(|e| TmuxError::CommandFailed { command: "...", stderr: format!("task join error: {e}") })?
```

## Proposed change
Create a helper in the core crate that encapsulates the spawn_blocking + output checking pattern:

```rust
async fn run_tmux<T>(
    command_name: &str,
    target: &str,
    cmd: impl Into<TmuxCommand>,
    on_success: impl FnOnce(&std::process::Output) -> T + Send + 'static,
) -> Result<T, TmuxError>
```

Each operation becomes a thin wrapper calling this helper with its specific command and success handler.

This is related to but independent of #40 (TmuxAdapter trait). This issue eliminates boilerplate within the current architecture; #40 adds the abstraction boundary for testability.

## Acceptance criteria
- [ ] Helper function defined in core crate (private to crate)
- [ ] At least 3 operation modules refactored to use it
- [ ] No behavior change in public API
- [ ] Reduced line count per operation module
