# Extract reusable tmux command execution pattern to reduce per-operation boilerplate

All 13 domain operation files repeat an identical pattern: clone inputs → spawn_blocking → Tmux::with_command → .output() → check status → from_stderr → map join error. This creates ~20 lines of boilerplate per operation.

## Problem
Every operation file (new_session.rs, kill_session.rs, list_windows.rs, etc.) contains:
1. Clone string parameters into owned values
2. tokio::task::spawn_blocking(move || { ... })
3. Tmux::with_command(Cmd::new()...).output().map_err(|e| TmuxError::CommandFailed { command: "cmd-name", stderr: e.to_string() })?
4. if !output.status.success() { return Err(TmuxError::from_stderr(...)) }
5. .await.map_err(|e| TmuxError::CommandFailed { command: "cmd-name", stderr: format!("task join error: {e}") })?

This is repeated 13 times with only the command construction and result parsing differing.

## What to do
Extract a generic command executor in the core crate:

async fn run_tmux<F, T>(command_name: &str, target: &str, build_and_run: F) -> Result<T, TmuxError>
where F: FnOnce() -> Result<(std::process::Output, T), TmuxError> + Send + 'static, T: Send + 'static

Or a simpler approach: a helper that takes a tmux_interface command, runs it in spawn_blocking, checks status, and returns stdout/stderr. Each operation calls the helper and only implements the command-specific logic.

## Acceptance criteria
- Each operation file is reduced to its unique logic (command construction + result parsing)
- The spawn_blocking + status check + error mapping pattern exists in exactly one place
- Adding a new tmux operation requires only the command-specific code
