# Add domain health model with component-level readiness checks

The domain has \`is_available()\` and \`server_info()\` for basic tmux health, but production deployments need richer health information: is the server responsive? Is it under memory pressure? Can it accept new sessions? This is a domain concern, not an API concern.

## Problem
- \`is_available()\` is binary — tmux is either running or not
- No information about tmux server health (session count, client count, uptime)
- API health endpoints (#25) need richer data from the domain
- Kubernetes readiness/liveness probes need granular health information
- No way to detect degraded states (server running but overloaded)

## Proposed solution

\`\`\`rust
pub struct HealthStatus {
    pub available: bool,
    pub version: String,
    pub session_count: u32,
    pub client_count: u32,
    pub uptime_seconds: Option<u64>,    // server start time → now
    pub server_pid: Option<u32>,
}

/// Comprehensive health check of the tmux server.
pub async fn health_check() -> HealthStatus
\`\`\`

This is a composite query that gathers data from multiple tmux commands (server-info, list-sessions, list-clients) into a single domain model. It never fails — if tmux is unreachable, it returns \`HealthStatus { available: false, ... }\`.

API layers use this for:
- REST: \`/health\` endpoint with JSON body
- gRPC: health service reporting
- GraphQL: health query with fields

## Scope
- New \`health.rs\` module in \`tmux-gateway-core\`
- \`HealthStatus\` domain model
- \`health_check\` composite operation
- Add to \`TmuxCommands\` trait
- Replaces ad-hoc health checking in API layers
