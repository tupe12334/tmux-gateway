# Add domain-level idempotency support for create operations

## Problem

Domain create operations (new_session, new_window) fail if the target already exists. There’s no idempotent “create if not exists” variant. Clients that want safe retries or declarative state must handle duplicates themselves, leading to race conditions between check-and-create.

## Current state

````rust
pub async fn new_session(name: &str) -> Result<String, TmuxError>
// Fails with CommandFailed (or SessionAlreadyExists after #43) if session exists
````

Clients must: list sessions → check if name exists → create if not. This has a TOCTOU race.

## Proposed change

Add idempotent variants in the domain core:

````rust
pub async fn ensure_session(name: &str) -> Result<TmuxSession, TmuxError>
pub async fn ensure_window(session: &str, name: &str) -> Result<TmuxWindow, TmuxError>
````

These create the resource if it doesn’t exist, or return the existing one if it does. Atomicity is best-effort (tmux doesn’t support transactions), but the domain function handles the retry internally.

Depends on: #39 (session existence check), #43 (SessionAlreadyExists error).

## Acceptance criteria

* [ ] ensure_session creates or returns existing session
* [ ] ensure_window creates or returns existing window
* [ ] No TOCTOU race in the domain implementation
* [ ] Added to TmuxCommands trait
