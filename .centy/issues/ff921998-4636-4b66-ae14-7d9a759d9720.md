# Add domain concurrency guard with optimistic versioning

## Problem

Multiple API clients can concurrently modify the same tmux session, window, or pane. Read-modify-write patterns (e.g., "list windows, then rename the active one") are susceptible to lost updates when another client modifies the target between the read and write steps.

The domain layer has no mechanism to detect or prevent concurrent modification conflicts.

## Proposed change

Add optimistic concurrency control as a pure domain concept in `tmux-gateway-core`:

```rust
/// A fingerprint of the current state of a tmux resource, used to detect concurrent modifications.
#[derive(Clone, Debug, PartialEq, Eq, Hash)]
pub struct ResourceVersion {
    target: String,
    fingerprint: String,  // hash of relevant state (name, window count, pane count, etc.)
    observed_at: i64,
}

impl ResourceVersion {
    /// Pure: compute fingerprint from a session snapshot
    pub fn from_session(session: &TmuxSession) -> Self;

    /// Pure: compute fingerprint from a window snapshot
    pub fn from_window(target: &str, window: &TmuxWindow) -> Self;

    /// Pure: check if a current snapshot still matches this version
    pub fn matches(&self, current: &ResourceVersion) -> bool;
}

#[derive(Debug)]
pub struct ConcurrentModification {
    pub target: String,
    pub expected: ResourceVersion,
    pub actual: ResourceVersion,
}
```

Add a `TmuxError::ConcurrentModification` variant so APIs can return HTTP 409 / gRPC ABORTED when a version mismatch is detected.

Domain operations that accept an optional `expected_version: Option<ResourceVersion>` parameter can verify the resource hasn't changed before proceeding. The version check itself is a pure comparison — the imperative shell fetches the current state and calls the pure check.

## Why this matters

- Multi-client scenarios (dashboard + CLI + automation) are the primary use case for a gateway
- Without concurrency control, last-write-wins causes silent data loss
- Issue #93 (batch operations) and #46 (composite operations) amplify the risk of concurrent modification

## Acceptance criteria

- [ ] `ResourceVersion` value object with pure `from_session`, `from_window` constructors
- [ ] `ConcurrentModification` error type
- [ ] `TmuxError::ConcurrentModification` variant with HTTP 409 / gRPC ABORTED mapping
- [ ] Pure `matches()` comparison — no I/O in the domain layer
- [ ] Unit tests for version computation, matching, and mismatch detection
- [ ] Domain operations accept optional version parameter (non-breaking change)
